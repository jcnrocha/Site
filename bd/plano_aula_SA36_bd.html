📖 Revisão do 3º Trimestre SA 30 → SA 34 · 5 blocos
SA 30 LEFT JOIN · RIGHT JOIN · Anti-join
Conceito-chave
  • INNER JOIN — apenas linhas com correspondência nos dois lados
  • LEFT JOIN — todas as linhas da esquerda + NULL para as sem par na direita
  • RIGHT JOIN — todos da direita + NULL para os sem par à esquerda
  • Anti-join — LEFT JOIN + WHERE direita.col IS NULL → linhas sem correspondência
⚠️ LEFT JOIN com filtro WHERE direita.col = 'x' elimina os NULLs — comportamento igual ao INNER JOIN.
Exemplo — clientes sem nenhuma locação
-- clientes que nunca locaram (anti-join)
SELECT c.nome
FROM cliente c
LEFT JOIN locacao l
  ON c.cpf = l.cpf_cliente
WHERE l.id_locacao IS NULL;
SA 31 Subqueries — WHERE IN · Scalar · FROM derivada
Tipos de subquery
  • WHERE IN (subquery) — filtra com lista dinâmica de valores
  • WHERE NOT IN — exclui os encontrados pela subquery (cuidado: NULL anula tudo)
  • Scalar — retorna um único valor; usada em comparações com >, =
  • FROM derivada — subquery como tabela virtual no FROM; obrigatório ter alias
⚠️ NOT IN com NULL na subquery retorna zero linhas — sempre use WHERE col IS NOT NULL dentro da subquery.
Exemplo — filmes mais caros que a média
-- scalar: valor acima da média do catálogo
SELECT titulo, valor_diaria
FROM filme
WHERE valor_diaria > (
  SELECT AVG(valor_diaria)
  FROM filme
);
SA 32 Views — CREATE VIEW · CREATE OR REPLACE · DROP
Comandos essenciais
  • CREATE VIEW nome AS SELECT … — cria a view; error se já existir
  • CREATE OR REPLACE VIEW — cria ou substitui sem error
  • DROP VIEW IF EXISTS nome — remove sem error se não existir
  • View não armazena dados — é uma query nomeada; cada SELECT recalcula
  • Permite simplificar queries longas e ocultar colunas sensíveis
⚠️ Uma view sobre tabelas com JOINs pesados executa o JOIN a cada consulta — não é um atalho de performance, é um atalho de legibilidade.
Exemplo — view de histórico de locações
CREATE OR REPLACE VIEW vw_historico AS
SELECT c.nome, f.titulo,
       l.data_locacao
FROM locacao l
INNER JOIN cliente c
  ON l.cpf_cliente = c.cpf
INNER JOIN filme f
  ON l.id_filme = f.id_filme;
SA 33 UPDATE · DELETE · Transações (BEGIN / COMMIT / ROLLBACK)
Regras de ouro do DML
  • Sempre use WHERE no UPDATE e no DELETE — sem WHERE, afeta TODOS os registros
  • BEGIN inicia a transação · COMMIT confirma · ROLLBACK desfaz
  • Padrão seguro: BEGIN → SELECT (verificar) → UPDATE/DELETE → SELECT (conferir) → COMMIT
  • MySQL em modo autocommit=1 (padrão) executa cada comando como sua própria transação
⚠️ Nunca execute UPDATE ou DELETE em produção sem BEGIN primeiro — mesmo que esteja com pressa.
Padrão seguro — UPDATE com transação
BEGIN;
-- verificar antes de alterar
SELECT * FROM cliente
WHERE cpf = '111.222.333-44';
UPDATE cliente
SET cidade_cliente = 'Umuarama'
WHERE cpf = '111.222.333-44';
-- conferir o resultado
SELECT * FROM cliente
WHERE cpf = '111.222.333-44';
COMMIT; -- ou ROLLBACK
SA 34 Documentação — -- e /* */ · O que comentar
Regras da documentação
  • -- ignora até o final da linha · ideal para notas curtas ao lado do código
  • /* */ ignora o bloco inteiro · usado no cabeçalho e para desativar código
  • Comente o porquê, nunca o o quê — o código já diz o quê
  • Comentário desatualizado é pior que nenhum — ele mente
  • Cabeçalho de bloco é obrigatório em scripts entregues
⚠️ Comentário que repete o código em palavras ("seleciona o nome da tabela cliente") não agrega valor e polui o script.
Cabeçalho padrão obrigatório
/*
  Banco: locadora
  Script: [descrição do que o script faz]
  Propósito: [para que serve / quem usa]
  Autor: [nome] · Data: dd/mm/aaaa
*/

-- regra de negócio não-óbvia: comentar aqui
SELECT nome FROM cliente
WHERE ativo = 1; -- 0=bloqueado p/ marketing
Mapa de sintaxe — referência rápida para a prova
ComandoSintaxe mínima obrigatóriaPonto de atenção
INNER JOIN FROM a INNER JOIN b ON a.fk = b.pk Só retorna linhas com correspondência nos dois lados
LEFT JOIN FROM a LEFT JOIN b ON a.fk = b.pk Mantém todos de "a"; NULL quando não há par em "b"
Anti-join LEFT JOIN … WHERE b.pk IS NULL Filtra apenas os sem correspondência à direita
WHERE IN WHERE col IN (SELECT … FROM …) Subquery deve retornar uma única coluna
Scalar WHERE col > (SELECT AVG(col) FROM …) Subquery deve retornar exatamente um valor
FROM derivada FROM (SELECT … FROM …) alias Alias é obrigatório — sem ele dá erro de sintaxe
CREATE VIEW CREATE OR REPLACE VIEW nome AS SELECT … Não armazena dados; recalcula a cada SELECT
UPDATE seguro BEGIN; UPDATE t SET col=v WHERE pk=x; COMMIT; Sem WHERE: atualiza TODOS os registros
DELETE seguro BEGIN; DELETE FROM t WHERE pk=x; COMMIT; Sem WHERE: apaga TODOS os registros
📝 Dicas para a Prova — SA 37 · 25/11/2026 5 objetivas + 5 discursivas
📅 Prova em 25/11/2026 (SA 37) — banco de dados: Locadora (tabelas CLIENTE, FILME, ATOR, ELENCO, LOCACAO) · 5 questões objetivas + 5 discursivas · coberta: SELECT, JOINs, GROUP BY, Views, UPDATE/DELETE, Documentação
✔ Tópicos com maior peso
  • JOIN com ON correto — é o coração da prova
  • LEFT JOIN + IS NULL (anti-join) — cai quase sempre
  • GROUP BY + HAVING — diferença de WHERE vs HAVING
  • CREATE VIEW e consultar a VIEW depois
  • UPDATE e DELETE dentro de BEGIN/COMMIT
  • Identificar comentário útil vs comentário óbvio
✘ Erros mais comuns — evitar
  • Esquecer o ON no JOIN (erro de sintaxe)
  • Usar WHERE com função de agregação — usar HAVING
  • FROM derivada sem alias — erro obrigatório de sintaxe
  • NOT IN com coluna que pode ter NULL — retorna vazio
  • DROP VIEW sem IF EXISTS — erro se a view não existir
  • UPDATE/DELETE sem WHERE — altera/deleta tudo
💡 Dica final: nas questões discursivas, mostrar o código e explicar uma linha diferente — "usamos LEFT JOIN porque precisávamos manter os filmes sem locação na listagem; INNER JOIN os excluiria." Justificativas valem pontos mesmo que o código tenha um erro menor de sintaxe.
