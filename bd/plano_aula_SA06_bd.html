SQL — As Quatro Categorias da Linguagem
DDL
Data Definition Language
Define e modifica a estrutura do banco — tabelas, colunas, índices e restrições.
CREATE ALTER DROP TRUNCATE
DML
Data Manipulation Language
Manipula os dados — inserção, atualização e exclusão de registros nas tabelas.
INSERT UPDATE DELETE
DQL
Data Query Language
Realiza consultas ao banco — recupera dados conforme critérios e filtros definidos.
SELECT
DCL
Data Control Language
Controla o acesso ao banco — concessão e revogação de permissões por usuário.
GRANT REVOKE
-- DDL: criar tabela
CREATE TABLE clientes (id INT PRIMARY KEY, nome VARCHAR(100), email VARCHAR(100));

-- DML: inserir e atualizar
INSERT INTO clientes (id, nome) VALUES (1, 'Ana Silva');
UPDATE clientes SET email = 'ana@email.com' WHERE id = 1;

-- DQL: consultar
SELECT nome, email FROM clientes WHERE id = 1;
NoSQL — Quatro Tipos de Banco Não Relacional
📄
Documento
Dados em formato JSON/BSON. Flexibilidade de esquema — cada documento pode ter campos diferentes.
ExemplosMongoDB · CouchDB
🔑
Chave-Valor
Estrutura simples de par chave → valor. Altíssima velocidade — ideal para cache e sessões.
ExemplosRedis · DynamoDB
📊
Coluna
Dados organizados por colunas em vez de linhas. Otimizado para analytics e grandes volumes.
ExemplosCassandra · HBase
🕸️
Grafo
Entidades como nós e relacionamentos como arestas. Ideal para redes sociais e sistemas de recomendação.
ExemplosNeo4j · Amazon Neptune
SQL × NoSQL — Comparativo Direto
Critério SQL (Relacional) NoSQL (Não Relacional)
EstruturaTabelas com esquema fixo e definidoDocumentos, pares, colunas ou grafos — esquema flexível
EscalabilidadeVertical — aumenta o servidorHorizontal — adiciona mais servidores
ConsistênciaACID — Atomicidade, Consistência, Isolamento, DurabilidadeBASE — Basically Available, Soft state, Eventual consistency
FlexibilidadeBaixa — alterações de esquema são complexasAlta — campos podem variar por registro
ConsultasSQL padronizado e poderoso com JOINsAPIs próprias de cada banco, sem padrão único
MaturidadeAlta — décadas de evolução e suporteVariável — alguns maduros, outros emergentes
Quando Usar Cada Abordagem
SQL
Banco de Dados Relacional
  • Dados estruturados com esquema bem definido
  • Relacionamentos complexos entre entidades
  • Transações ACID obrigatórias (financeiro, saúde)
  • Consultas complexas com múltiplos JOINs
  • Volume de dados gerenciável em um servidor
NoSQL
Banco de Dados Não Relacional
  • Alto volume de dados e escalabilidade horizontal
  • Esquema variável ou dinâmico por registro
  • Dados não estruturados: JSON, mídia, logs
  • Alta disponibilidade com baixa latência
  • Redes sociais, IoT, jogos, recomendação

💡 SQL ou NoSQL? — Votação em 5 Cenários
⏱️ Duração: 15 minutos
👥 Agrupamento: turma inteira — votação individual
🎯 Objetivo: desenvolver raciocínio de escolha técnica por contexto
1
Apresentação do cenário (1 min/rodada) — Professor projeta o cenário e pergunta: "Para este projeto você usaria SQL ou NoSQL? Por quê?"
2
Votação (1 min) — Cada aluno vota: SQL ou NoSQL. Professor registra os votos no quadro.
3
Justificativa e consolidação (2 min) — Professor escolhe alunos de cada lado para justificar. Em seguida apresenta a resposta mais adequada com justificativa técnica.
Cenário Resposta adequada Justificativa técnica
Sistema de controle financeiro de um banco — transações, contas, saldos e extratos SQL Transações ACID obrigatórias, relacionamentos complexos, integridade de dados crítica
Plataforma de streaming que recomenda filmes com base no histórico de cada usuário NoSQL — Grafo Modelagem de conexões entre usuários, filmes e preferências é naturalmente um grafo
Sistema de cadastro de clientes, produtos e pedidos de uma loja virtual de médio porte SQL Relacionamentos bem definidos, estrutura estável, volume gerenciável
Aplicativo de chat com milhões de mensagens simultâneas e histórico por conversa NoSQL — Documento Alto volume, esquema flexível (texto, imagem, vídeo), escalabilidade horizontal
Sistema acadêmico com alunos, professores, disciplinas, notas e frequências SQL Relacionamentos múltiplos, integridade referencial, consultas complexas com JOINs
💡 Ponto pedagógico: a resposta correta depende sempre do contexto — volume de dados, tipo de relacionamento, necessidade de transações e flexibilidade de esquema. Não existe escolha universalmente correta.
