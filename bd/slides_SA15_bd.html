Conceito central da aula

🎯 O Que É Modelagem de Dados?

Modelagem de dados é o processo de planejar, antes de programar, como as informações de um sistema vão ser organizadas, relacionadas e armazenadas.

O que ela define A estrutura, não o código
Quais são as "coisas" do sistema (entidades), o que elas guardam (atributos) e como se conectam (relacionamentos) — tudo isso antes de qualquer SQL.
Por que existe Evitar retrabalho caro
Corrigir um erro de estrutura depois que o banco já está populado é muito mais caro do que corrigir no papel.

Resumindo: modelar é desenhar a planta antes de levantar a parede.

Analogia · engenharia civil

🏗️ A Planta e a Obra

Nenhum engenheiro começa a construir uma casa sem planta. O banco de dados segue a mesma lógica: primeiro o projeto, depois a construção.

📐 A Planta
Representa a modelagem: o projeto no papel, pensado, revisado e aprovado antes de qualquer material ser comprado.
🏠 A Obra
Representa o banco de dados implementado: a casa de verdade, construída seguindo exatamente o que a planta definiu.

No projeto da Biblioteca: o DER, o esquema lógico e o script SQL são as plantas; o banco funcionando é a obra entregue.

Uma confusão comum

⚖️ Modelo de Dados × Banco de Dados

📐 Modelo de Dados
→ É o projeto, o desenho da estrutura
→ Existe no papel, em diagramas ou texto
→ Independente de SGBD específico
→ Pode ser revisado sem custo de dados
🗄️ Banco de Dados
→ É a implementação real, em funcionamento
→ Existe dentro de um SGBD (MySQL, PostgreSQL...)
→ Já guarda dados reais de verdade
→ Mudar a estrutura agora tem custo e risco

Conclusão: você passa a maior parte do tempo do projeto trabalhando no modelo — o banco em si é só a etapa final.

Visão geral · as três fases

📊 As Três Fases da Modelagem

FaseO que representaArtefato
ConceitualEntidades e relacionamentos, sem tecnologiaDER
LógicaTabelas, chaves primárias e estrangeiras, sem SQLEsquema lógico
FísicaImplementação real no SGBD, com tipos e restriçõesScript SQL

Destaque: cada fase é mais detalhada e mais "amarrada à tecnologia" que a anterior — você nunca pula direto para o SQL.

Fase 1 de 3 · o nível mais abstrato

📐 Fase Conceitual

A fase Conceitual representa o sistema em alto nível: quais entidades existem e como elas se relacionam — sem pensar em tabelas, tipos ou SGBD.

O que aparece
Entidades (ex: LIVRO, MEMBRO), seus atributos principais e os relacionamentos entre elas (ex: MEMBRO empresta LIVRO).
O que NÃO aparece
Tipos de dados, tamanhos de campo, SQL ou qualquer detalhe de implementação — isso vem nas próximas fases.

Artefato desta fase: o DER (Diagrama Entidade-Relacionamento) — você vai construir o seu na SA 16 e na SA 17.

Fase 2 de 3 · estrutura de tabelas

📋 Fase Lógica

A fase Lógica traduz o DER em estrutura de tabelas: nomes de tabela, colunas, chaves primárias e estrangeiras — ainda sem código SQL.

-- notação textual do esquema lógico
FILME (id_filme PK, titulo, genero, ano)
LOCACAO (id_locacao PK, id_cliente FK, id_filme FK, data_locacao)

Artefato desta fase: o esquema lógico — você vai escrevê-lo na SA 18, convertendo o DER da Biblioteca.

Fase 3 de 3 · implementação real

⚙️ Fase Física

A fase Física é onde o esquema lógico se transforma em código SQL real, executável em um SGBD — com tipos de dados e restrições completos.

📋 Lógico (antes)
FILME (id_filme PK,
  titulo, genero, ano)
⚙️ Físico (depois)
CREATE TABLE filme (
  id_filme INT PRIMARY KEY,
  titulo VARCHAR(150) NOT NULL,
  ano INT
);

Artefato desta fase: o script SQL completo — você vai produzi-lo na SA 19, fechando as três fases do projeto da Biblioteca.

Exemplo guiado · sistema de locadora de filmes

📋 Um Mesmo Sistema, Três Representações

Veja como a relação "cliente aluga filme" evolui pelas três fases, ficando mais detalhada a cada passo.

01
Conceitual: CLIENTE e FILME são entidades; "aluga" é o relacionamento entre elas — sem nenhum detalhe técnico.
02
Lógica: nasce a tabela LOCACAO com id_cliente FK e id_filme FK para representar esse relacionamento.
03
Física: CREATE TABLE locacao (...) com tipos de dados, PRIMARY KEY e REFERENCES reais no SGBD.
O custo de não planejar

⚖️ Sem Modelagem × Com Modelagem

✗ Banco criado sem planejar
✗ Dados repetidos e inconsistentes
✗ Relacionamentos perdidos ou ambíguos
✗ Mudanças exigem reescrever tabelas inteiras
✗ Erros só aparecem quando o sistema já está em uso
✔ Banco planejado em fases
✔ Estrutura revisada antes de qualquer dado existir
✔ Relacionamentos claros e documentados
✔ Mudanças no papel custam tempo, não dados perdidos
✔ Problemas aparecem e são corrigidos no diagrama

Conclusão: é exatamente isso que você vai sentir comparando seu script da SA 14 com o banco modelado da SA 19.
