O que é uma View?

Uma View é uma consulta SQL armazenada no banco de dados com um nome. Quando consultada, ela é executada dinamicamente — sempre reflete os dados atuais das tabelas base. Não armazena dados, apenas armazena a definição da consulta.

📐 Simplificação
Encapsula JOINs e subqueries complexas. Quem consulta a View usa um SELECT simples sem precisar conhecer a estrutura interna do banco.
🔒 Segurança
Permite expor apenas as colunas necessárias, ocultando dados sensíveis como CPF, senha ou salário. O usuário acessa a View mas não tem permissão na tabela base.
♻️ Reutilização
A mesma View pode ser usada em diferentes consultas, relatórios e sistemas. Mudanças na query base são feitas em um único lugar — todos que usam a View se beneficiam.
Anatomia do CREATE VIEW
comando DDL
CREATE VIEW vw_historico AS
início do SELECT interno
SELECT c.nome AS cliente,
       f.titulo AS filme,
       f.genero,
colunas expostas pela view
       l.data_locacao
tabelas base com JOINs
FROM cliente c
INNER JOIN locacao l ON c.cpf = l.cpf_cliente
INNER JOIN filme f ON l.id_filme = f.id_filme
ordenação padrão da view
ORDER BY l.data_locacao DESC;
Consultando e filtrando uma View
-- View como tabela: sintaxe idêntica ao SELECT normal
SELECT * FROM vw_historico;

-- aplicando WHERE sobre a View
SELECT * FROM vw_historico
WHERE genero = 'Ação'
ORDER BY cliente;
View de segurança — ocultando dados sensíveis
-- expõe nome e cidade; oculta CPF e email
CREATE VIEW vw_clientes_pub AS
SELECT nome, cidade_cliente
FROM cliente;
Atualizar e remover Views
-- recriar sem precisar apagar antes
CREATE OR REPLACE VIEW vw_historico AS
SELECT c.nome AS cliente, f.titulo, f.genero, l.data_locacao
FROM cliente c
INNER JOIN locacao l ON c.cpf = l.cpf_cliente
INNER JOIN filme f ON l.id_filme = f.id_filme;

-- remover permanentemente
DROP VIEW vw_historico;
DROP VIEW IF EXISTS vw_historico; -- sem erro se não existir
💡 View não armazena dados: cada vez que você faz SELECT * FROM vw_historico, o banco executa a consulta interna naquele momento. Se um novo registro for inserido na LOCACAO, a View automaticamente o refletirá na próxima consulta — sem necessidade de atualizar nada.

1
Crie uma View chamada vw_historico que mostre o nome do cliente, o título do filme, o gênero e a data da locação — simplificando o JOIN de três tabelas usado nos relatórios da locadora
CLIENTE + LOCACAO + FILME Simplificação
Gabarito esperado
CREATE VIEW vw_historico AS
SELECT c.nome AS cliente,
       f.titulo,
       f.genero,
       l.data_locacao
FROM cliente c
INNER JOIN locacao l ON c.cpf = l.cpf_cliente
INNER JOIN filme f ON l.id_filme = f.id_filme;

-- usando a View:
SELECT * FROM vw_historico ORDER BY data_locacao DESC;
2
Crie uma View chamada vw_clientes_pub que exponha apenas o nome e a cidade dos clientes, ocultando CPF e e-mail para uso em relatórios públicos
CLIENTE Segurança
Gabarito esperado
CREATE VIEW vw_clientes_pub AS
SELECT nome, cidade_cliente
FROM cliente;

-- quem acessa vw_clientes_pub nunca vê cpf nem email
SELECT * FROM vw_clientes_pub ORDER BY nome;
3
Crie uma View chamada vw_total_cliente que mostre o nome de cada cliente e o total de locações realizadas, incluindo apenas clientes que já fizeram pelo menos uma locação
CLIENTE + LOCACAO Reutilização
Gabarito esperado
CREATE VIEW vw_total_cliente AS
SELECT c.nome, COUNT(*) AS total_locacoes
FROM cliente c
INNER JOIN locacao l ON c.cpf = l.cpf_cliente
GROUP BY c.nome;

-- reutilizando: top 3 clientes
SELECT * FROM vw_total_cliente ORDER BY total_locacoes DESC LIMIT 3;
4
Usando a vw_historico criada na rodada 1, escreva uma consulta que mostre apenas os registros do gênero 'Drama', ordenados pelo nome do cliente
vw_historico + WHERE sobre View
Gabarito esperado
-- ponto chave: WHERE e ORDER BY aplicados sobre a View, não sobre as tabelas
SELECT cliente, titulo, data_locacao
FROM vw_historico
WHERE genero = 'Drama'
ORDER BY cliente;
💡 Ponto pedagógico — rodada 4: reforçar que aplicar WHERE sobre uma View é equivalente a aplicar WHERE dentro da query interna dela — o banco otimiza isso automaticamente (predicate pushdown). O aluno não precisa reescrever a View para cada filtro diferente.
⚠️ View não é substituta de índice: Views não melhoram performance por si só — elas executam a query definida toda vez que são consultadas. Se a query interna for lenta, a View também será. Para performance, a solução são índices nas tabelas base, não Views.
🔁 CREATE OR REPLACE VIEW: apresentar como boa prática ao reconstruir Views em produção — evita o DROP + CREATE, mantém permissões existentes sobre a View e é mais seguro em ambientes compartilhados.
