🏆 Objetivos de Aprendizagem 5 objetivos

Ao final da aula, o aluno será capaz de:

  • 1Compreender o que é o modelo físico e como ele se diferencia do modelo lógico
  • 2Traduzir um esquema lógico em comandos SQL utilizando CREATE TABLE com tipos de dados e restrições
  • 3Aplicar corretamente as restrições de integridade: PRIMARY KEY, FOREIGN KEY, NOT NULL, UNIQUE, DEFAULT e CHECK
  • 4Escolher o tipo de dado adequado para cada coluna com base na natureza da informação
  • 5Identificar a relação direta entre cada decisão do modelo lógico e o código SQL gerado no modelo físico

Restrições de integridade
PRIMARY KEY
Identifica unicamente cada linha da tabela.
id INT PRIMARY KEY
FOREIGN KEY
Garante integridade referencial entre tabelas.
FK (col) REFERENCES TAB(col)
NOT NULL
Impede valor vazio em coluna obrigatória.
nome VARCHAR(100) NOT NULL
UNIQUE
Garante que não há valores duplicados na coluna.
email VARCHAR(100) UNIQUE
DEFAULT
Define valor padrão quando nenhum é fornecido.
ativo BOOLEAN DEFAULT true
CHECK
Valida uma condição antes de inserir ou atualizar.
CHECK (preco > 0)
AUTO_INCREMENT
Gera automaticamente valores únicos crescentes.
id INT AUTO_INCREMENT
⚠️ Ordem de criação: tabelas sem FOREIGN KEY devem ser criadas antes das que dependem delas — o SGBD não aceita referenciar uma tabela que ainda não existe.

🎮 Dinâmica Obrigatória
✅ Certo ou Errado? — Round Físico
⏱️ Duração: 15 minutos
👥 Agrupamento: turma inteira
🖊️ Materiais: slides com os blocos SQL · quadro branco para votos

Reativação da dinâmica da Aula 01, agora voltada ao modelo físico — fixar restrições de integridade e escolha de tipos de dados por análise crítica de código SQL com erros intencionais.

1
Apresentação do bloco (1 min por rodada)
O professor exibe um CREATE TABLE no slide — código limpo, sem marcação de erro.
2
Votação (1 min)
A turma decide: CERTO ✅ ou ERRADO ❌. O professor registra os votos no quadro.
3
Justificativa obrigatória (2 min)
Quem votou ❌ aponta a linha com problema e explica o erro. Sem justificativa, o ponto não conta.
4
Reveal e consolidação (1 min)
O professor exibe a versão corrigida e reforça a regra técnica envolvida.
RodadaBloco SQLProblema
1 CREATE TABLE cliente (
  cpf VARCHAR(11),
  nome VARCHAR(150) NOT NULL,
  email VARCHAR(100)
); 		Ausência de PRIMARY KEY — nenhuma coluna identifica unicamente o registro
2 CREATE TABLE locacao (
  id INT PRIMARY KEY AUTO_INCREMENT,
  id_cliente VARCHAR(11),
  data_locacao DATE DEFAULT
); 		DEFAULT sem valor definido — sintaxe incompleta
3 CREATE TABLE elenco (
  id_filme INT NOT NULL,
  id_ator INT NOT NULL,
  FOREIGN KEY (id_filme)
  REFERENCES filmes(id_filme)
); 		Nome da tabela referenciada errado — filmes deveria ser filme
4 CREATE TABLE produto (
  id INT PRIMARY KEY,
  preco CHAR(10),
  estoque INT DEFAULT 0 NOT NULL
); 		Tipo inadequado — CHAR(10) não permite operações numéricas; preco deveria ser DECIMAL
⚠️ Atenção — Rodada 4: o bloco está sintaticamente correto, mas semanticamente errado. O objetivo é que os alunos percebam que erros de design não geram mensagem de erro no SGBD, mas causam problemas sérios de integridade e desempenho.
