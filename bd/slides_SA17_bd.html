Conceito central da aula

🎯 O Que É um Relacionamento no DER?

Um relacionamento conecta duas entidades, representando como elas se associam no mundo real. No DER, é desenhado como um losango entre as entidades.

Nome do relacionamento Sempre no infinitivo
Nomeado como verbo: Realiza, Pertence, Possui — descreve a ação que conecta as entidades.
Grau do relacionamento Quantas entidades envolve
Unário (entidade com ela mesma), binário (duas entidades — o mais comum) ou ternário (três entidades).

No projeto da Biblioteca: MEMBRO realiza EMPRESTIMO e EMPRESTIMO envolve LIVRO são exemplos de relacionamentos binários.

Antes dos três tipos

🎯 O Que É Cardinalidade?

Cardinalidade define quantas instâncias de uma entidade podem se relacionar com quantas instâncias de outra — e é exatamente essa regra que vira a estrutura do banco depois.

Por que ela importa
Define se vai existir uma coluna FK simples ou uma tabela associativa inteira no modelo lógico — a cardinalidade não é só desenho, é decisão de estrutura.
Como ela é lida
Sempre nos dois sentidos: "um MEMBRO pode ter quantos EMPRESTIMOS?" e "um EMPRESTIMO pertence a quantos MEMBROS?"

Regra de negócio: a cardinalidade é definida pelo que o sistema real PRECISA permitir — não é uma escolha técnica aleatória.

Os três tipos · visão geral

📊 1:1, 1:N e N:N

CardinalidadeSignificadoExemplo
1 : 1Cada instância de A se relaciona com no máximo uma de B, e vice-versaFuncionário ↔ Crachá
1 : NUma instância de A se relaciona com várias de B; cada B pertence a um ACliente → Pedidos
N : NVárias instâncias de A se relacionam com várias de BAluno ↔ Disciplinas

Destaque: o N:N é especial — ele vai exigir uma tabela nova (associativa) quando você converter para o modelo lógico na SA 18.

Tipo 1 de 3

1️⃣ Cardinalidade 1:1 — Um para Um

O tipo mais restritivo: cada instância de uma entidade se relaciona com no máximo uma instância da outra, nos dois sentidos.

Exemplo clássico
FUNCIONARIO ↔ CRACHA: um funcionário tem exatamente um crachá; um crachá pertence a exatamente um funcionário.
Quando aparece
É o tipo mais raro. Geralmente indica que as duas entidades poderiam até ser uma só — mas existem razões para separá-las.

No projeto da Biblioteca: não há um 1:1 evidente — a maioria das relações reais envolve "vários", como você vai ver nos próximos dois tipos.

Tipo 2 de 3 · o mais comum

🔢 Cardinalidade 1:N — Um para Muitos

O tipo mais frequente em sistemas reais: uma instância do lado "1" se relaciona com várias do lado "N", mas cada uma do lado "N" pertence a apenas uma do lado "1".

Exemplo genérico
CLIENTE → PEDIDOS: um cliente pode ter vários pedidos; cada pedido pertence a um único cliente.
No modelo lógico
Esse tipo gera uma simples coluna de chave estrangeira (FK) na tabela do lado "N" — sem precisar de tabela nova.

No projeto da Biblioteca: MEMBRO realiza EMPRESTIMO é 1:N — um membro pode ter vários empréstimos, mas cada empréstimo é de um único membro.

Tipo 3 de 3 · o mais complexo

🔀 Cardinalidade N:N — Muitos para Muitos

Aqui várias instâncias de A se relacionam com várias de B, nos dois sentidos — esse tipo não pode ser representado com uma FK simples.

✗ Não funciona assim
✗ Uma FK em LIVRO apontando para AUTOR
✗ Um livro só poderia ter 1 autor
✗ Perderíamos livros com múltiplos autores
✔ Solução correta
✔ Criar uma tabela associativa nova
✔ LIVRO_AUTOR guarda os dois IDs
✔ Cada linha é uma combinação válida

No projeto da Biblioteca: LIVRO é escrito por AUTOR é N:N — um livro pode ter vários autores, e um autor pode escrever vários livros.

Mais um detalhe da cardinalidade

📊 Participação Total × Parcial

TipoSignificadoExemplo
TotalToda instância da entidade DEVE participar do relacionamentoTodo pedido pertence a um cliente
ParcialA instância PODE participar ou não do relacionamentoUm cliente pode não ter pedidos

No DER: participação total é desenhada com linha dupla; participação parcial, com linha simples.

Caso especial

🎯 Relacionamento Pode Ter Atributo Próprio

Às vezes uma informação não pertence a nenhuma das duas entidades — pertence ao relacionamento entre elas.

Por que isso acontece
A informação só existe NO MOMENTO em que a relação ocorre — não antes, não depois, e não faz sentido em nenhuma das entidades isoladas.
Exemplo no projeto
data_saida e data_devolucao não pertencem a MEMBRO nem a LIVRO — pertencem ao próprio EMPRESTIMO.

Dica: quando um relacionamento 1:N ou N:N precisa guardar dados próprios, ele frequentemente se torna uma entidade própria no modelo lógico.

Habilidade prática essencial

📋 Como Ler a Cardinalidade nos Dois Sentidos

Toda cardinalidade tem duas perguntas — uma para cada lado do relacionamento. Veja o passo a passo usando MEMBRO e EMPRESTIMO.

01
Pergunte do lado A para B: "Um MEMBRO pode estar em quantos EMPRESTIMOS?" → resposta: vários (N).
02
Pergunte do lado B para A: "Um EMPRESTIMO pertence a quantos MEMBROS?" → resposta: um só (1).
03
Combine as duas respostas: MEMBRO (1) realiza EMPRESTIMO (N) → cardinalidade 1:N confirmada.
