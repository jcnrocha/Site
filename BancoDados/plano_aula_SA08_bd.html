📚 Conteúdo da Aula CREATE TABLE · Tipos · Restrições
Sintaxe do CREATE TABLE
-- estrutura básica
CREATE TABLE nome_tabela (
    nome_coluna  TIPO_DADO  RESTRIÇÕES,
    nome_coluna  TIPO_DADO  RESTRIÇÕES,
    ...
);

-- exemplo completo: tabela de clientes
CREATE TABLE clientes (
    id          INT               PRIMARY KEY AUTO_INCREMENT,
    nome       VARCHAR(150)    NOT NULL,
    email      VARCHAR(100)    UNIQUE NOT NULL,
    cpf        CHAR(11)        UNIQUE,
    ativo      BOOLEAN         DEFAULT TRUE,
    criado_em  DATETIME        DEFAULT CURRENT_TIMESTAMP
);

Restrições de Integridade
PRIMARY KEY
Identifica unicamente cada linha da tabela. Não aceita NULL nem valores duplicados. Cada tabela deve ter exatamente uma.
id INT PRIMARY KEY
AUTO_INCREMENT
Gera um valor numérico sequencial automaticamente para cada novo registro. Usado junto com PRIMARY KEY.
id INT PRIMARY KEY AUTO_INCREMENT
NOT NULL
Impede que a coluna receba valor nulo. O dado é obrigatório em toda operação de inserção.
nome VARCHAR(100) NOT NULL
UNIQUE
Garante que não existam valores duplicados em uma coluna. Permite NULL (a menos que NOT NULL também esteja presente).
email VARCHAR(100) UNIQUE
DEFAULT valor
Define o valor inserido automaticamente quando nenhum valor é fornecido para aquela coluna.
ativo BOOLEAN DEFAULT TRUE  estoque INT DEFAULT 0
Boas Práticas
Usar letras minúsculas e underscore nos nomes — nome_tabela, data_nascimento
Nomes no singular para tabelas — cliente, não clientes (padrão do setor)
Usar DECIMAL(p,s) para valores monetários, nunca FLOAT ou CHAR
Toda tabela deve ter uma PRIMARY KEY — preferir AUTO_INCREMENT para simplicidade
Definir NOT NULL em colunas obrigatórias do negócio desde o início
DEFAULT deve sempre ter um valor explícito — nunca escrever apenas DEFAULT sozinho

🔍 Certo ou Errado? — Análise de Código SQL
⏱️ Duração: 15 minutos
👥 Agrupamento: turma inteira
🎯 Objetivo: identificar erros de sintaxe e tipo em código CREATE TABLE
1
Apresentação do bloco (1 min/rodada) — Professor exibe o CREATE TABLE no slide — sem marcação de erro.
2
Votação (1 min) — A turma decide: ✅ CERTO ou ❌ ERRADO. Professor registra os votos no quadro.
3
Justificativa obrigatória (2 min) — Quem votou ❌ aponta a linha com problema e explica o erro. Sem justificativa, o ponto não conta.
4
Reveal e consolidação (1 min) — Professor exibe a versão corrigida e reforça a regra técnica envolvida.
1
CREATE TABLE cliente — sem chave primária
❌ Errado
CREATE TABLE cliente (
    nome  VARCHAR(150) NOT NULL,
    email VARCHAR(100) UNIQUE,
    cpf   CHAR(11)
);
Problema identificado
Ausência de PRIMARY KEY — nenhuma coluna identifica unicamente o registro. Toda tabela deve ter uma chave primária definida.
2
CREATE TABLE produto — tipo inadequado para preço
❌ Errado
CREATE TABLE produto (
    id      INT PRIMARY KEY AUTO_INCREMENT,
    nome    VARCHAR(100) NOT NULL,
    preco   CHAR(10),
    estoque INT DEFAULT 0
);
Problema identificado
Tipo inadequado — preco CHAR(10) não permite operações numéricas (soma, média, comparação). O correto é DECIMAL(10,2).
3
CREATE TABLE pedido — sintaxe e tipos corretos
✅ Correto
CREATE TABLE pedido (
    id            INT                         PRIMARY KEY AUTO_INCREMENT,
    data_pedido    DATE                        NOT NULL DEFAULT (CURRENT_DATE),
    valor_total    DECIMAL(10,2)             NOT NULL
);
Tudo certo
Sintaxe, tipos e restrições estão adequados. DECIMAL(10,2) para valor monetário, DATE para data e DEFAULT (CURRENT_DATE) com valor explícito.
4
CREATE TABLE funcionario — DEFAULT sem valor
❌ Errado
CREATE TABLE funcionario (
    id      INT PRIMARY KEY,
    nome    VARCHAR(150) NOT NULL,
    salario DECIMAL(8,2) DEFAULT,
    ativo   BOOLEAN DEFAULT TRUE
);
Problema identificado
DEFAULT sem valor definido — sintaxe inválida na coluna salario. O correto seria DEFAULT 0.00 ou remover a restrição.
💡 Ponto pedagógico: a rodada 3 é intencionalmente correta. Saber reconhecer um código bem escrito é tão importante quanto saber identificar erros — desenvolve segurança e senso crítico no aluno.
