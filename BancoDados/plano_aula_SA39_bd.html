✅ Correção Comentada — Questões Objetivas (SA 37 e SA 38) Erros mais comuns por questão
ℹ️ As provas SA 37 e SA 38 cobrem os mesmos cinco tópicos do 3º Trimestre — as questões são diferentes, mas os conceitos avaliados são os mesmos. A correção a seguir se aplica a ambas as provas.
Q1 SA37: C  ·  SA38: A LEFT JOIN — comportamento com e sem correspondência
Conceito correto
LEFT JOIN mantém todas as linhas da tabela à esquerda. Quando não há correspondência à direita, as colunas dessa tabela ficam como NULL. O anti-join (WHERE direita.col IS NULL) isola exatamente essas linhas.
Erro mais comum
Confundir LEFT JOIN com INNER JOIN. Muitos alunos marcaram a alternativa que falava "clientes que realizaram locações" — que seria o resultado de um INNER JOIN. A ideia de manter linhas sem par à direita é o ponto central que diferencia os dois.
Q2 SA37: C  ·  SA38: B NOT IN com NULL / Subquery WHERE IN
Conceito correto
SA37: NOT IN com NULL na subquery não gera erro — ele simplesmente retorna zero linhas, silenciosamente. Isso é um bug difícil de detectar. Solução: WHERE col IS NOT NULL dentro da subquery.

SA38: a subquery de WHERE IN deve retornar uma única coluna — a alternativa D com SELECT * geraria erro.
Erro mais comum
Acreditar que SQL avisa quando NOT IN recebe NULL ("retorna erro de execução"). SQL não avisa — silenciosamente devolve zero linhas. Esse é um dos bugs mais perigosos do SQL porque não quebra a query, apenas retorna resultados errados.
Q3 SA37: B  ·  SA38: C GROUP BY + HAVING / CREATE OR REPLACE VIEW
Conceito correto
SA37: WHERE filtra linhas individuais antes do agrupamento — não aceita funções de agregação. HAVING filtra grupos formados pelo GROUP BY.

SA38: CREATE OR REPLACE VIEW não gera erro se a view já existir — é a forma segura de usar em scripts que podem rodar mais de uma vez.
Erro mais comum
SA37: usar WHERE COUNT(*) > x — gera erro de sintaxe. Muitos alunos ainda colocam o filtro de agregação no WHERE por hábito. SA38: confundir CREATE OR REPLACE com "apaga e recria os dados" — a view nunca armazena dados.
Q4 SA37: B  ·  SA38: C Views — armazenamento / DELETE sem WHERE
Conceito correto
SA37: VIEW é uma query nomeada — não armazena dados. Cada SELECT sobre ela reexecuta o SELECT interno. DROP VIEW remove a definição, nunca as tabelas base.

SA38: DELETE sem WHERE remove todos os registros da tabela. Não há confirmação automática no MySQL — daí a importância de BEGIN/COMMIT antes de qualquer DELETE.
Erro mais comum
SA37: marcar que VIEW armazena dados fisicamente — mito persistente. SA38: marcar que o banco pede confirmação, o que não acontece em nenhum SGBD padrão. A proteção real é o próprio BEGIN/COMMIT, que é sempre responsabilidade do programador.
Q5 SA37: C  ·  SA38: C Documentação — comentário que NÃO segue boas práticas / /* */ para blocos
Conceito correto
SA37: comentário que repete o código em palavras ("seleciona o nome e o e-mail da tabela cliente") não agrega valor — o código já diz isso. Um bom comentário explica o propósito ou uma regra de negócio não-óbvia.

SA38: /* */ é a única sintaxe padrão para comentário de bloco em SQL. O -- só desativa uma linha.
Erro mais comum
SA37: selecionar o cabeçalho /* Script: … */ como resposta errada — muitos não reconheceram o formato de cabeçalho de bloco como boa prática. SA38: marcar # como válido para blocos — funciona em MySQL para linha, mas não é o padrão ANSI e não desativa múltiplas linhas de uma vez.
📊 Padrão de erros da turma — o que monitorar: erros de conceito (confundir LEFT JOIN com INNER JOIN, acreditar que VIEW armazena dados) precisam de reforço antes do próximo ano. Erros de sintaxe (WHERE com COUNT, SELECT * em subquery) tendem a desaparecer naturalmente com prática. Na SA 40, priorize revisitar os conceitos, não a sintaxe.

Padrões de erro mais observados nas discursivas
Q6 LEFT JOIN + COUNT Clientes com total de locações incluindo zero
Ponto crítico
O erro mais importante aqui é usar COUNT(*) em vez de COUNT(l.id_locacao). O COUNT(*) conta o próprio NULL como 1, fazendo clientes sem locação aparecerem com 1 em vez de 0. A coluna do JOIN — não o asterisco — é o segredo.
Código correto resumido
SELECT c.nome, COUNT(l.id_locacao) AS total
FROM cliente c LEFT JOIN locacao l
  ON c.cpf = l.cpf_cliente
GROUP BY c.cpf, c.nome
ORDER BY total DESC;
Q7 Subquery scalar / WHERE IN Subquery como filtro — scalar (SA37) e WHERE IN (SA38)
Ponto crítico
SA37 (scalar): a subquery com AVG() deve estar entre parênteses e retorna um único valor — se o aluno a colocar sem parênteses ou com múltiplas colunas, dá erro.

SA38 (WHERE IN): a subquery interna precisa de JOIN entre LOCACAO e FILME — muitos escreveram apenas SELECT cpf_cliente FROM locacao WHERE genero = 'Ação', o que falha pois LOCACAO não tem a coluna genero.
Código correto resumido
-- SA37: scalar
WHERE valor_diaria > (SELECT AVG(valor_diaria) FROM filme)

-- SA38: WHERE IN com JOIN interno
WHERE cpf IN (SELECT l.cpf_cliente FROM locacao l
  INNER JOIN filme f ON l.id_filme = f.id_filme
  WHERE f.genero = 'Ação')
Q8 · Q9 · Q10 VIEW · Transação · Doc CREATE VIEW · UPDATE/DELETE com BEGIN · Cabeçalho + comentários úteis
Q8 — Erro mais comum
Criar a view sem o JOIN que junta as tabelas necessárias — ou usar CREATE VIEW sem OR REPLACE. Outro erro frequente: esquecer o SELECT sobre a view após criá-la.
Q9 · Q10 — Erros mais comuns
Q9: omitir o BEGIN ou esquecer o SELECT de verificação antes do COMMIT. Alguns alunos escreveram COMMIT + ROLLBACK juntos — só um dos dois deve ser executado.

Q10: manter algum comentário óbvio e/ou escrever o cabeçalho como comentário de linha (--) em vez de bloco (/* */). O cabeçalho com quatro campos (banco, propósito, autor, data) era obrigatório.

A linha do aprendizado — como tudo se conecta
Começo: entendemos o problema — como organizar dados do mundo real em tabelas sem redundância, com relacionamentos claros. O DER é a planta da casa.
Construção: transformamos o DER em código SQL — CREATE TABLE com PKs e FKs. O modelo saiu do papel e entrou no banco de dados real.
Dados: populamos o banco com INSERT e aprendemos a consultá-lo — SELECT, WHERE, ORDER BY. As funções de agregação transformaram dados brutos em informações (contagens, médias, totais).
Relacionamentos: o INNER JOIN conectou tabelas, respondendo perguntas que cruzavam mais de uma entidade. A base do pensamento relacional.
Visão completa: LEFT JOIN + subqueries permitiram perguntas sofisticadas — "quem está faltando?", "quem está acima da média?". Views tornaram o banco mais organizado. Transações tornaram as operações seguras.
Profissionalismo: a documentação fechou o ciclo — código que só funciona não basta. Código que pode ser mantido, compreendido e evoluído por outros é o padrão profissional.
🔭 Perspectivas — O Que Vem no Próximo Nível Curiosidade, não ansiedade
SQL ainda mais poderoso — próximas fronteiras
  • Stored Procedures e Functions — lógica de negócio dentro do banco: parâmetros, condicionais, loops em SQL
  • Triggers — código que executa automaticamente quando dados mudam (ex.: log de auditoria a cada UPDATE)
  • Índices — como acelerar queries lentas; por que uma mesma query pode ser 100× mais rápida com um índice bem posicionado
  • Transações avançadas — níveis de isolamento, concorrência, deadlocks
  • Window Functions — RANK, ROW_NUMBER, LAG/LEAD — análises sem GROUP BY
Além do SQL — o ecossistema completo
  • NoSQL — MongoDB, Redis, Cassandra — quando tabelas não são a melhor escolha: documentos, chave-valor, grafos
  • SQL no back-end — integrar o banco a uma aplicação Python, Node.js ou Java; ORMs e queries dinâmicas
  • Modelagem avançada — Data Warehouse, OLAP, estrela e floco de neve para Business Intelligence
  • Segurança — permissões granulares, SQL Injection, dados sensíveis em repouso
  • Cloud databases — MySQL na AWS, Azure SQL, Supabase — banco como serviço gerenciado
💡 Mensagem de encerramento: tudo o que foi construído neste ano — modelar, criar, popular, consultar, proteger e documentar um banco de dados — é a base que sustenta praticamente todo sistema de software no mercado. Quem entende banco de dados entende o coração de qualquer aplicação.
