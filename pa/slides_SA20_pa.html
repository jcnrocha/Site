🧪 O Que É um Teste Unitário?

Teste unitário verifica UMA unidade isolada do código — geralmente um único método — de forma automatizada, sem banco de dados, rede ou arquivos externos.

Características Isolado · Automatizado · Rápido
Isolado: sem banco, rede ou arquivo externo · Automatizado: roda com um clique, sempre igual · Rápido: cada teste executa em milissegundos
Benefícios Detectar · Especificar · Refatorar
Detecta bugs imediatamente após introduzi-los · funciona como especificação executável do método · permite refatorar com segurança

Regra: depois de alterar o código, se todos os testes passarem, você sabe que nada quebrou — isso é a rede de segurança

B6 — Técnicas de Programação

📐 O Padrão AAA — Arrange · Act · Assert

O padrão AAA divide cada teste em três fases claras — tornando-o legível, organizado e fácil de manter por qualquer membro da equipe.

01
Arrange — Preparar — Criar o objeto a ser testado e definir as entradas. Ex: Calculadora c = new Calculadora(); double n1 = 8.0, n2 = 6.0;
02
Act — Agir — Chamar o método sendo testado. Uma única ação por teste. Ex: double resultado = c.calcularMedia(n1, n2);
03
Assert — Verificar — Confirmar que o resultado obtido é igual ao esperado. Verde: passou. Vermelho: falhou com detalhe. Ex: assertEquals(7.0, resultado);
B6 — Técnicas de Programação

💻 Teste Manual × Código AAA

Verificar manualmente é lento, inconsistente e não escala — o padrão AAA transforma a verificação em código automatizado e repetível.

✗ Antes — Verificação Manual
// Verificação manual — lento e incerto
Calculadora c = new Calculadora();
System.out.println(
  c.calcularMedia(8.0, 6.0));
// Verifico visualmente "7.0"
// Tenho que fazer isso TODA alteração
✔ Depois — Padrão AAA
@Test
void deveRetornar7QuandoNotas8e6() {
  Calculadora c = new Calculadora();
  // Act
  double r = c.calcularMedia(8.0, 6.0);
  // Assert
  assertEquals(7.0, r);
} // Verde = passou · Vermelho = detalhe

Resultado: com AAA o teste documenta o comportamento esperado e verifica automaticamente — sem intervenção humana a cada mudança

B6 — Teste Unitário

✅ Assertions Mais Usadas

Assertion Quando usar Exemplo
assertEquals Valor obtido deve ser igual ao esperado assertEquals(7.0, resultado)
assertTrue / False Resultado de condição booleana assertTrue(ehMaiorDeIdade(18))
assertNull / NotNull Objeto nulo ou não nulo assertNotNull(cliente.buscar())
assertThrows Método deve lançar exceção específica assertThrows(Exception.class, () -> ...)
assertNotEquals Valor obtido deve ser diferente do comparado assertNotEquals(0.0, resultado)

O método de teste falha se a assertion não for satisfeita — a mensagem indica esperado × obtido, apontando exatamente o que está errado

B6 — Boas Práticas de Teste

🏷️ Convenções de Nomenclatura

Um bom nome de teste é uma especificação executável — lendo apenas o nome, qualquer pessoa entende o que o método deve fazer e em qual situação.

Padrão deve[Comportamento]Quando[Condição]
deveRetornar7_QuandoNotas8e6() · deveRetornarFalse_QuandoIdadeFor17() · deveLancarExcecao_QuandoNomeDuplicado()
Estrutura Classe · Pasta · Sufixo
Classe de teste: CalculadoraTest.java (sufixo Test) · Pasta: src/test/java · Uma classe de teste por classe de produção

Convenção: classe de teste tem o mesmo nome da classe original com o sufixo Test — exemplo: CalculadoraTest

B6 — Técnicas de Programação

⚖️ Teste Manual × Teste Automatizado

✗ Teste Manual
✗ Lento — rodar o programa e verificar visualmente
✗ Inconsistente — depende do testador e do momento
✗ Não escala — inviável conforme o sistema cresce
✗ Sem documentação — ninguém sabe o que foi verificado
✗ Reexecução trabalhosa — refazer tudo a cada alteração
✔ Teste Automatizado
✔ Rápido — milissegundos, centenas de testes de uma vez
✔ Consistente — a mesma verificação toda execução
✔ Escala — quanto mais testes, mais seguro o sistema
✔ Documentado — o código de teste é a especificação
✔ Reexecução: um clique — a qualquer momento

Conclusão: testes automatizados são o que permite que sistemas cresçam sem se tornarem frágeis — o custo de escrever é menor que o custo de corrigir bugs em produção

B6 — Prática na IDE

▶️ Executando Testes na IDE

No VS Code com a extensão de teste, cada método @Test recebe um botão de play — o resultado aparece imediatamente com detalhe sobre o que passou ou falhou.

01
Instalar JUnit 5 — adicionar a dependência junit-jupiter no pom.xml do projeto Maven
02
Criar a classe de teste — CalculadoraTest.java na pasta src/test/java
03
Escrever o método de teste — anotar com @Test e seguir o padrão AAA (Arrange · Act · Assert)
04
Executar — clicar no botão de play ao lado do método ou rodar mvn test no terminal
05
Interpretar — Verde: passou · Vermelho: falhou, com esperado × obtido indicando exatamente o que está errado
B6 — Boas Práticas

🔒 O Teste como Rede de Segurança

Testes unitários são a rede de segurança do desenvolvedor — não impedem erros, mas garantem que você saiba imediatamente quando cometeu um.

Durante o desenvolvimento Escrever · Alterar · Refatorar
Cada novo método vem com seu teste · alterar código não quebra o que funcionava · confiança para refatorar sem medo de efeitos colaterais
No ciclo profissional Qualidade · Velocidade · Documentação
Projetos com testes têm menos bugs em produção · equipes entregam mais rápido com mais confiança · testes funcionam como documentação viva do sistema

Regra prática: se você não consegue testar um método isoladamente, é sinal de que ele está fazendo coisas demais — hora de refatorar

Atividade prática em grupo

🎮 Dinâmica — Escreve o Teste

🧪
Escreve o Teste — Padrão AAA na Prática · 15 minutos · grupos de 4 a 5 alunos
01
Apresentação (1 min) — O professor projeta a assinatura e o comportamento esperado de um método. Os grupos escrevem o teste unitário no padrão AAA — sem ver a implementação
02
Execução em grupo (8 min) — Os grupos escrevem os testes na IDE ou na ficha, cobrindo ao menos dois cenários: cenário feliz (entrada válida) e cenário de borda (valor no limite ou inválido)
03
Correção ao vivo (6 min) — O professor apresenta a implementação dos métodos · os grupos executam seus testes · verificam se passam ou falham · debatem os cenários não cobertos

Itens da dinâmica: 4 métodos com assinatura e comportamento esperado — grupos escrevem pelo menos dois cenários (feliz e borda) para cada método

Escreve o Teste · casos práticos

📋 Casos para Analisar

Método Comportamento esperado Cenários sugeridos
boolean ehMaiorDeIdade(int idade) Retorna true se idade >= 18 18 (limite) · idade = 17 · idade = 0
double calcularDesconto(double preco, double pct) Retorna preço com desconto aplicado preco=100, pct=10 → 90.0 · pct=0 → preço original
String classificarNota(double nota) Retorna "Aprovado", "Recuperação" ou "Reprovado" nota=7.0 · nota=5.0 · nota=4.9
boolean cpfValido(String cpf) Retorna true se CPF tem 11 dígitos numéricos CPF válido · com letras · menos dígitos

TDD na prática: escrever o teste antes de ver a implementação força pensar no comportamento esperado — não na implementação. Empresas que adotam isso têm menos bugs em produção.
