Slide 05 · UPDATE
UPDATE — Atualizando Registros no Banco
Java (JDBC)
String sql = "UPDATE alunos SET nota = ? WHERE id = ?"; try (Connection conn = Conexao.getConexao(); PreparedStatement ps = conn.prepareStatement(sql)) { ps.setDouble(1, novaNota); ps.setInt(2, idAluno); int linhas = ps.executeUpdate(); if (linhas > 0) { conn.commit(); System.out.println("Nota atualizada!"); } else { System.out.println("Aluno não encontrado."); } } catch (SQLException e) { System.err.println(e.getMessage()); }
Python (sqlite3)
sql = "UPDATE alunos SET nota = ? WHERE id = ?" cursor.execute(sql, (nova_nota, id_aluno)) conexao.commit() if cursor.rowcount > 0: print("Nota atualizada!") else: print("Aluno não encontrado.")
cursor.rowcount em Python e executeUpdate() em Java retornam o número de linhas afetadas — sempre verifique se é maior que zero antes de exibir "sucesso"
Sintaxe SQL: UPDATE tabela SET coluna1 = ?, coluna2 = ? WHERE condição · Atualizar múltiplas colunas: SET nota = ?, turma = ? — cada coluna separada por vírgula.
Slide 06 · Perigo
UPDATE sem WHERE — O Erro Mais Perigoso do CRUD
❌ Sem WHERE — catástrofe
// Intenção: atualizar nota de um aluno // Esqueceu o WHERE... String sql = "UPDATE alunos SET nota = ?"; ps.setDouble(1, 10.0); ps.executeUpdate(); conn.commit(); // RESULTADO: nota de TODOS os alunos // virou 10.0 — irreversível após o commit
✅ Com WHERE — correto
String sql = "UPDATE alunos SET nota = ? WHERE id = ?"; ps.setDouble(1, 10.0); ps.setInt(2, idAluno); int linhas = ps.executeUpdate(); conn.commit(); // Apenas o aluno com aquele ID foi atualizado // linhas == 1 confirma que era o esperado
Regra absoluta: UPDATE sem WHERE atualiza todos os registros da tabela. No mercado, operações de UPDATE e DELETE em produção exigem revisão por um segundo desenvolvedor, backup prévio e confirmação explícita. Conferir o WHERE duas vezes antes de executar é hábito obrigatório.
Slide 07 · DELETE
DELETE — Removendo Registros do Banco
Java (JDBC)
String sql = "DELETE FROM alunos WHERE id = ?"; try (Connection conn = Conexao.getConexao(); PreparedStatement ps = conn.prepareStatement(sql)) { ps.setInt(1, idAluno); int linhas = ps.executeUpdate(); if (linhas > 0) { conn.commit(); System.out.println("Aluno removido!"); } else { System.out.println("Aluno não encontrado."); } } catch (SQLException e) { System.err.println(e.getMessage()); }
Python (sqlite3)
sql = "DELETE FROM alunos WHERE id = ?" cursor.execute(sql, (id_aluno,)) conexao.commit() if cursor.rowcount > 0: print("Aluno removido!") else: print("Aluno não encontrado.")
⚠️ Python: ao passar apenas um valor como tupla, não esqueça a vírgula: (id_aluno,) — sem a vírgula não é uma tupla.
Sintaxe SQL: DELETE FROM tabela WHERE condição · O DELETE não aceita lista de colunas — ele remove a linha inteira. Sempre use Prepared Statement para o valor da condição WHERE.
Slide 08 · Perigo
DELETE sem WHERE — Operação Irreversível
❌ Sem WHERE — apaga tudo
// Intenção: remover um aluno específico // Esqueceu o WHERE... String sql = "DELETE FROM alunos"; ps.executeUpdate(); conn.commit(); // RESULTADO: TODOS os alunos removidos // Após commit, não há desfazer // Tabela vazia — dados perdidos
✅ Com WHERE — seguro
String sql = "DELETE FROM alunos WHERE id = ?"; ps.setInt(1, idAluno); int linhas = ps.executeUpdate(); if (linhas > 0) { conn.commit(); System.out.println("Removido!"); } // Apenas o aluno com aquele ID foi removido
DELETE sem WHERE remove todos os registros da tabela. Diferente do UPDATE, que pode ser corrigido atualizando os dados de volta, o DELETE destrói os dados permanentemente após o commit. Em ambientes profissionais, DELETE em produção exige backup obrigatório antes de executar.
Slide 09 · Linhas Afetadas
Linhas Afetadas — Verificar Antes de Confirmar Sucesso
executeUpdate() em Java e cursor.rowcount em Python retornam o número de linhas que foram alteradas pela operação. Zero significa que o WHERE não encontrou nenhum registro correspondente — o sistema NÃO deve exibir "sucesso" nesse caso.
RetornoSignificaO que fazer
0 Nenhum registro encontrado com o WHERE informado Não chamar commit · exibir "registro não encontrado"
1 Exatamente um registro foi alterado — esperado para UPDATE/DELETE por ID Chamar commit · exibir "operação realizada"
> 1 Múltiplos registros alterados — pode ser intencional (ex: desconto geral) ou erro de WHERE Verificar se a quantidade era esperada · confirmar só se fizer sentido
Java: int linhas = ps.executeUpdate(); if (linhas == 0) { System.out.println("Nenhum registro afetado."); } else { conn.commit(); }
Python: cursor.execute(sql, (valor, id_aluno)) if cursor.rowcount == 0: print("Nenhum registro afetado.") else: conexao.commit()
Slide 10 · Exclusão Lógica
Exclusão Lógica × DELETE Físico
DELETE físico — permanente
// Remove o registro da tabela para sempre String sql = "DELETE FROM alunos WHERE id = ?"; ps.setInt(1, idAluno); ps.executeUpdate(); conn.commit(); // Após commit: dado irrecuperável // Sem backup? Dado perdido para sempre
Exclusão lógica — reversível
// Marca o registro como inativo // O dado continua no banco, mas "desaparece" String sql = "UPDATE alunos SET ativo = false WHERE id = ?"; ps.setInt(1, idAluno); ps.executeUpdate(); conn.commit(); // Consultas filtram apenas ativos: // SELECT * FROM alunos WHERE ativo = true
🔄
Reversível
Exclusão lógica pode ser desfeita: basta atualizar ativo = true — o dado nunca saiu do banco
📊
Histórico
Mantém o histórico de registros removidos — útil para auditorias, relatórios e rastreabilidade
🏢
Uso Profissional
A maioria dos sistemas corporativos usa exclusão lógica — clientes, pedidos e usuários raramente são deletados de verdade
Slide 11 · Transações
Transações — Tudo ou Nada
Transação é um conjunto de operações executado como uma unidade indivisível: ou todas as operações são confirmadas, ou nenhuma delas é. O banco de dados nunca fica em um estado parcialmente modificado.
🔓
setAutoCommit(false)
Desativa o commit automático — operações ficam em espera
⚙️
Operação 1
Executada mas não confirmada — ainda pode ser desfeita
⚙️
Operação 2
Executada mas não confirmada — parte da mesma transação
💾
commit()
Confirma as duas juntas — banco atualizado de forma consistente
↩️
rollback()
Se algo falhar — desfaz tudo desde o início. Banco volta ao estado original
rollback() não desfaz commit já confirmado. Ele só age sobre as operações pendentes desde o último commit. Uma vez que o commit foi executado, as alterações são permanentes.
Slide 12 · O Problema
Transferência Bancária — Sem Transação
Cenário: transferir R$ 500 da conta A para a conta B. São duas operações: débito na origem e crédito no destino. O que acontece se ocorrer um erro entre as duas?
Sequência sem transação
// Operação 1: débitar da conta A String sql1 = "UPDATE contas SET saldo = saldo - ? WHERE id = ?"; ps1.executeUpdate(); conn.commit(); // commit individual ← ERRO // Erro ocorre aqui: banco cai, conexão perde... // Operação 2: creditar na conta B // NUNCA ACONTECE String sql2 = "UPDATE contas SET saldo = saldo + ? WHERE id = ?";
Resultado
💸
Débito ocorreu
R$ 500 saíram da conta A — o commit individual confirmou o débito
❌
Crédito não ocorreu
R$ 500 nunca chegaram à conta B — o dinheiro simplesmente desapareceu
💥
Banco inconsistente
O sistema está em estado inválido — dados corrompidos, sem como reverter
Slide 13 · A Solução
Transferência Bancária — Com Transação
Java — com transação
conn.setAutoCommit(false); // início da transação try { // Operação 1: débito ps1.setDouble(1, valor); ps1.setInt(2, contaOrigem); ps1.executeUpdate(); // Operação 2: crédito ps2.setDouble(1, valor); ps2.setInt(2, contaDestino); ps2.executeUpdate(); conn.commit(); // confirma as DUAS juntas System.out.println("Transferência concluída!"); } catch (SQLException e) { conn.rollback(); // desfaz TUDO se qualquer uma falhar System.err.println("Erro: transferência cancelada."); } finally { conn.setAutoCommit(true); // restaura comportamento padrão conn.close(); }
Resultado com transação
✅
Tudo funciona
Débito E crédito confirmados juntos pelo commit — banco sempre consistente
↩️
Erro no meio
rollback() desfaz o débito automaticamente — conta A fica intacta, nenhuma conta é afetada
🔐
Princípio: Tudo ou Nada
A transação garante que duas operações dependentes sejam indivisíveis — nunca uma sem a outra
Slide 14 · Prática
commit e rollback — Estrutura Completa com try/catch/finally
Java
conn.setAutoCommit(false); try { // operações de banco ps.executeUpdate(); conn.commit(); // confirma se tudo ok } catch (SQLException e) { conn.rollback(); // desfaz se falhou System.err.println(e.getMessage()); } finally { conn.setAutoCommit(true); conn.close(); // sempre fecha }
Python
try: cursor.execute(sql1, params1) cursor.execute(sql2, params2) conexao.commit() # confirma tudo except Exception as e: conexao.rollback() # desfaz tudo print(f"Erro: {e}") finally: cursor.close() conexao.close()
💾
commit()
Confirma permanentemente todas as operações da transação — executado no bloco try se tudo correu bem
↩️
rollback()
Desfaz todas as operações pendentes — executado no catch quando ocorre qualquer erro
🔒
finally
Restaura setAutoCommit(true) e fecha a conexão — sempre executado, com ou sem erro
Slide 15 · CRUD Completo
CRUD Completo — Visão Consolidada das 4 Operações
OperaçãoSQLMétodo Javacommit?Risco sem WHERE
INSERT INSERT INTO tabela (col) VALUES (?) executeUpdate() ✓ Sim Não se aplica
SELECT SELECT col FROM tabela WHERE cond executeQuery() ✗ Não Retorna todos os registros
UPDATE UPDATE tabela SET col = ? WHERE id = ? executeUpdate() ✓ Sim Atualiza TODOS os registros
DELETE DELETE FROM tabela WHERE id = ? executeUpdate() ✓ Sim Remove TODOS os registros
Padrão comum às 4 operações: sempre usar Prepared Statement · sempre verificar linhas afetadas antes de exibir "sucesso" · sempre fechar conexão no finally · usar transação quando duas ou mais operações dependem uma da outra.
Slide 16 · Boas Práticas
Boas Práticas — CRUD Completo
✓ FAZ
  • Sempre incluir WHERE em UPDATE e DELETE — conferir duas vezes
  • Verificar linhas afetadas antes de exibir mensagem de sucesso
  • Usar transação quando duas ou mais operações dependem uma da outra
  • Chamar rollback() no catch — desfaz tudo em caso de erro
  • Considerar exclusão lógica (ativo=false) em vez de DELETE físico
  • Registrar em log todas as operações de UPDATE e DELETE
✗ EVITA
  • UPDATE ou DELETE sem WHERE — altera ou remove todos os registros
  • Exibir "sucesso" sem verificar se rowcount > 0
  • Fazer commit individual em operações que dependem umas das outras
  • Esquecer rollback() no catch — banco fica em estado inconsistente
  • Usar DELETE físico em dados que podem precisar de recuperação futura
  • Ignorar o retorno de executeUpdate() — ele carrega informação vital
Slide 17 · Dinâmica
Operação Segura? — Análise Crítica de Cenários
🛡️
Dinâmica em Grupo · 15 minutos · Grupos de 4–5 alunos
Classificar cada cenário: Operação Segura · Com Risco · Incorreta — e justificar a decisão
CenárioClassificaçãoRisco / Correção
DELETE FROM alunos sem WHERE Incorreta — crítico Remove todos os alunos — sempre usar WHERE
UPDATE alunos SET nota = 10.0 WHERE id = ? com Prepared Statement Segura Atualiza apenas o aluno com o ID informado — uso correto
Dois INSERTs em tabelas relacionadas sem transação — o segundo falha Com Risco Banco fica inconsistente — usar transação com rollback no catch
DELETE com rowcount == 0 sem tratamento — sistema exibe "removido com sucesso" Com Risco Nenhum registro foi removido — verificar rowcount antes de exibir mensagem
UPDATE produtos SET preco = preco * 0.9 sem WHERE para desconto geral Segura (se intencional) Correto se o objetivo é atualizar todos — documentar e confirmar a intenção antes
rollback() chamado após commit bem-sucedido Incorreta rollback não desfaz commit confirmado — só age sobre operações ainda pendentes
Slide 18 · Resumo
O que aprendemos hoje — CRUD Completo
✏️
UPDATE
WHERE é obrigatório — sem ele, todos os registros são alterados. Verificar executeUpdate() > 0 antes de confirmar sucesso. Sempre usar Prepared Statement.
🗑️
DELETE
WHERE é obrigatório — sem ele, todos os registros são removidos de forma irreversível. Considere a exclusão lógica (ativo=false) para dados importantes.
🔐
Transações
setAutoCommit(false) agrupa operações. commit() confirma todas juntas. rollback() no catch desfaz tudo — garantindo o princípio de tudo ou nada.
✅
CRUD Completo
INSERT · SELECT · UPDATE · DELETE — as quatro operações fundamentais de qualquer sistema com banco de dados. A base de toda aplicação real está concluída.
