Modificando registros existentes

✏️ UPDATE — Como Atualizar Registros

UPDATE modifica registros existentes na tabela — sempre com WHERE para identificar quais alterar — sem WHERE, todos os registros são atualizados: erro gravíssimo em produção

FORMA CORRETA UPDATE com WHERE obrigatório
UPDATE alunos SET nota = ? WHERE id = ? · setDouble(1, nota) · setInt(2, id) · executeUpdate() retorna linhas afetadas · commit() confirma
⚠️ PERIGO SEM WHERE UPDATE alunos SET nota = 10.0
Sem WHERE → TODOS os alunos teriam nota 10 · erro irreversível em produção · sempre conferir o WHERE antes de executar qualquer UPDATE

Linhas afetadas: executeUpdate() retorna o número de registros modificados — valor 0 indica que o WHERE não encontrou nenhum registro correspondente

UPDATE na prática — Java

💻 UPDATE — Sem WHERE × Com WHERE

UPDATE sem WHERE é um dos erros mais graves em banco de dados — modifica todos os registros da tabela de forma que pode ser irreversível após o commit

✗ Sem WHERE — ERRADO
// ⚠️ Atualiza TODOS os alunos! String sql = "UPDATE alunos SET nota = ?"; PreparedStatement ps = con.prepareStatement(sql); ps.setDouble(1, nota); ps.executeUpdate(); con.commit(); // confirma o desastre
✔ Com WHERE — CORRETO
String sql = "UPDATE alunos SET nota = ?" + " WHERE id = ?"; PreparedStatement ps = con.prepareStatement(sql); ps.setDouble(1, nota); ps.setInt(2, id); int linhas = ps.executeUpdate(); if (linhas > 0) con.commit();

Boas práticas: sempre incluir WHERE · verificar as linhas afetadas antes de exibir mensagem de sucesso · conferir a condição duas vezes antes de executar

Removendo registros do banco

🗑️ DELETE — Como Remover Registros

DELETE remove registros da tabela — operação irreversível após commit — sempre usar WHERE para identificar exatamente o que será removido — sem WHERE, todos os registros são deletados

FORMA CORRETA DELETE com WHERE e verificação
DELETE FROM alunos WHERE id = ? · setInt(1, id) · verificar rowcount antes de commit · registrar (log) o DELETE · commit() confirma a remoção permanente
ALTERNATIVA MAIS SEGURA Exclusão Lógica
UPDATE alunos SET ativo = false WHERE id = ? · marca o registro como inativo em vez de removê-lo · dados ficam recuperáveis · recomendado para informações críticas

Exclusão lógica: campo ativo = false em vez de DELETE físico — dados removidos são difíceis de recuperar; a exclusão lógica permite desfazer a ação e auditar o histórico

DELETE na prática — Java

💻 DELETE — Sem WHERE × Com WHERE

DELETE sem WHERE remove todos os registros da tabela — operação destrutiva e irreversível após commit — responsável por alguns dos maiores incidentes de perda de dados em empresas

✗ Sem WHERE — ERRADO
// ⚠️ Remove TODOS os alunos! String sql = "DELETE FROM alunos"; PreparedStatement ps = con.prepareStatement(sql); ps.executeUpdate(); con.commit(); // catástrofe confirmada
✔ Com WHERE — CORRETO
String sql = "DELETE FROM alunos WHERE id = ?"; PreparedStatement ps = con.prepareStatement(sql); ps.setInt(1, id); int linhas = ps.executeUpdate(); if (linhas > 0) con.commit();

Verificar antes de confirmar: se linhas == 0, nenhum registro foi encontrado e o commit é desnecessário — exibir mensagem condizente com o resultado real

Verificando o resultado da operação

🔢 Linhas Afetadas — Confirmando o Resultado

executeUpdate() retorna o número de registros modificados — essa informação é fundamental para saber se a operação teve o efeito esperado antes de exibir mensagem de sucesso

JAVA · JDBC int linhas = ps.executeUpdate()
if (linhas > 0) → sucesso — commit() · if (linhas == 0) → WHERE não encontrou registro · if (linhas > 1) → WHERE afetou mais do que esperado — verificar condição
PYTHON · sqlite3 / mysql-connector cursor.rowcount
cursor.execute(sql, params) · if cursor.rowcount > 0: conexao.commit() · if cursor.rowcount == 0: WHERE não encontrou registro · verificar antes de exibir sucesso

0 linhas afetadas não é um erro de banco — é uma informação: o WHERE não encontrou nenhum registro correspondente — tratar explicitamente para não exibir "sucesso" incorretamente

Garantindo a integridade dos dados

🔐 Transações — commit e rollback

Uma transação é um conjunto de operações executado de forma atômica — tudo ou nada — se qualquer operação falhar, rollback desfaz todas as anteriores e o banco permanece consistente

CONFIRMAR commit() — torna permanente
Confirma todas as operações da transação · só após commit os dados são visíveis para outros usuários · sem commit, as alterações ficam em estado pendente — descartadas ao fechar a conexão
DESFAZER rollback() — desfaz tudo
Desfaz todas as operações desde o último commit · chamado no catch quando ocorre qualquer erro · não desfaz um commit já confirmado — rollback age apenas sobre operações ainda pendentes

setAutoCommit(false) em Java desativa o commit automático, colocando as operações na mesma transação · em Python o autocommit já é falso por padrão — commit e rollback funcionam diretamente

Cenário: transferência bancária

⚖️ Sem Transação × Com Transação

✗ Sem Transação — Risco
✗ Debita da conta A com sucesso
✗ Erro ocorre antes de creditar conta B
✗ Débito permanece — crédito não acontece
✗ Banco em estado inconsistente
✗ Dinheiro some sem rastreamento
✔ Com Transação — Seguro
✔ setAutoCommit(false) inicia a transação
✔ Debita A → erro → rollback() no catch
✔ O débito é completamente desfeito
✔ Se tudo certo → commit() confirma as duas
✔ Tudo ou nada — integridade garantida

Transações protegem operações dependentes entre si — se uma falha, rollback desfaz todas as anteriores e o banco permanece em um estado consistente e válido

Bloco completo com commit e rollback

📋 Transação — Passo a Passo

Use transações sempre que duas ou mais operações precisam ser executadas juntas — o banco garante que ou todas acontecem ou nenhuma acontece

01
Desativar autocommit — con.setAutoCommit(false) em Java · Python já tem autocommit desativado por padrão — pronto para a transação
02
Executar a primeira operação — ps1.executeUpdate() dentro do bloco try — UPDATE, DELETE ou INSERT da primeira parte da transação
03
Executar a segunda operação — ps2.executeUpdate() — outra operação dependente da primeira, ainda dentro do mesmo bloco try
04
commit() — confirmar — con.commit() no final do try — só após todas as operações terem sido executadas com sucesso
05
rollback() — desfazer em erro — con.rollback() no bloco catch — desfaz todas as operações pendentes e mantém o banco consistente
Visão consolidada das quatro operações

📊 CRUD Completo — Referência Final

Operação SQL + Método Java WHERE · commit
C — Create (INSERT)INSERT INTO t (c) VALUES (?) · executeUpdate()WHERE: não aplica · commit: Sim
R — Read (SELECT)SELECT c FROM t WHERE c = ? · executeQuery()WHERE: opcional · commit: Não
U — Update (UPDATE)UPDATE t SET c = ? WHERE id = ? · executeUpdate()WHERE: OBRIGATÓRIO · commit: Sim
D — Delete (DELETE)DELETE FROM t WHERE id = ? · executeUpdate()WHERE: OBRIGATÓRIO · commit: Sim

SELECT não modifica dados — não precisa de commit · UPDATE e DELETE sem WHERE são os erros mais perigosos em banco de dados — verificar sempre antes de executar
