As Três Frentes da Consolidação

🛠️
Técnicas de Programação
"As técnicas que transformam código funcional em código profissional — revisadas de uma só vez"
  • Formatação (6.1): indentação · formatOnSave · Prettier
  • Documentação (6.2): Javadoc/docstring · @param, @return
  • DRY (6.3): uma lógica em um lugar · classes utilitárias
  • Otimização (6.4): laços aninhados = O(n²) · break no laço
  • Depuração e logs (6.5–6.6): breakpoint · níveis DEBUG/INFO/WARN/ERROR
🧪
Testes no Código Existente
"Projeto fornecido com código funcional mas sem testes e com violações — a missão é analisar, testar e corrigir"
  • Padrão AAA · assertions · cenários de borda
  • Cobertura: identificar linhas não exercitadas
  • Testes como deveCalcularTotalComDescontoCorretamente()
  • Cenário de borda: assertThrows para desconto > 100%
🔄
Fluxo Profissional Integrado
"O fluxo de 8 passos que será o padrão do Projeto Final — do código ao commit"
  • 1. Código com nomes descritivos e formatação · 2. Documentar públicos
  • 3. DRY — extrair repetições · 4. Verificar complexidade
  • 5. Testes — feliz + borda · 6. Executar e corrigir falhas
  • 7. Verificar cobertura · 8. Commit com mensagem descritiva

Cinco Problemas Intencionais do Projeto GerenciadorPedidos

🏷️
Nomes não descritivos
calc, verificar, buscar — violação de nomenclatura
❓
Parâmetros sem significado
v e d — impossível saber o que representam sem ler tudo
🪜
Ifs aninhados
verificar pode ser simplificado para return total > 0 && itens > 0;
🐌
O(n²) desnecessário
buscar com dois for aninhados — um for com break resolve em O(n)
🚫
Sem doc e sem testes
Nenhum Javadoc/docstring · nenhum teste unitário no projeto

Testes que os Alunos Devem Criar para o Projeto

TesteO que verifica
deveCalcularTotalComDescontoCorretamente()100 com 10% de desconto = 90 — cenário feliz do cálculo
deveRetornarFalsoQuandoTotalForZero()Cenário de borda da aprovação — total inválido
deveRetornarFalsoQuandoItensForZero()Cenário de borda da aprovação — sem itens
deveBuscarIdExistenteERetornarIndice()Cenário feliz da busca no vetor
deveRetornarMenosUmQuandoIdNaoExistir()Cenário de borda da busca — item ausente
deveLancarExcecaoQuandoDescontoForMaiorQue100()Cenário de erro — desconto inválido com assertThrows
🔄 Fluxo integrado de desenvolvimento profissional (revisão): escrever o código com nomes descritivos e formatação → documentar os métodos públicos → aplicar DRY → verificar complexidade (substituir O(n²) quando possível) → escrever testes (feliz + borda) → executar e corrigir falhas → verificar cobertura e completar → commitar com mensagem descritiva. Este é o padrão esperado em todo o Projeto Final.

🎮 Dinâmica Auditoria de Código
🔍 Auditoria de Código — Code Review com Critérios Objetivos
⏱️ Duração: 15 minutos
👥 Agrupamento: grupos de 4 a 5 alunos
🎯 Objetivo: consolidar todas as técnicas de programação estudadas por meio de uma auditoria técnica de um trecho de código — simulando o processo real de code review que desenvolvedores realizam diariamente em equipes profissionais
1
Apresentação do código (2 min) — O professor projeta um trecho de código funcional mas com múltiplos problemas de técnica. Cada grupo recebe a ficha de auditoria e deve avaliar o código em 6 dimensões
2
Auditoria em grupo (8 min) — Os grupos analisam o código em cada dimensão, atribuem uma nota de 0 a 2 e descrevem o problema encontrado e a melhoria sugerida
3
Resultado e debate (5 min) — Cada grupo apresenta a nota geral e o problema mais crítico identificado. O professor consolida e discute quais problemas impactam mais na manutenção real
Dimensão da auditoriaO que avaliarNota (0–2)
FormataçãoIndentação, espaçamento, limite de linha0 · 1 · 2
NomenclaturaNomes de variáveis, métodos e classes descritivos0 · 1 · 2
DRYAusência de código repetido0 · 1 · 2
ComplexidadeLaços e estruturas sem O(n²) desnecessário0 · 1 · 2
DocumentaçãoJavadoc/docstring nos métodos públicos0 · 1 · 2
TestesPresença e qualidade dos testes unitários0 · 1 · 2
📊 Pontuação: 0 = problema grave · 1 = problema parcial · 2 = atende ao padrão · nota máxima: 12 pontos. Reforço pedagógico: este é o formato real de uma revisão de código em empresas — desenvolvedores avaliam código alheio com critérios técnicos objetivos, não com base em preferência pessoal. Saber dar e receber feedback técnico com clareza e respeito é uma das soft skills mais valiosas em times de desenvolvimento.
📝 Avaliação Questionário · 10 questões
Parte 1 — Questões Objetivas (5 questões)
  • Nível alto · 4 alternativas: A) B) C) D)
  • Gabarito alternado, sem repetição consecutiva de letras
  • Temas: qual técnica está sendo violada em um trecho, qual teste detectaria um bug dado, por que dois for aninhados são O(n²), a ordem correta do fluxo profissional, o que a cobertura indica e o que não garante
Parte 2 — Questões Discursivas (5 questões)
  • Cada questão possui título explicativo antes do enunciado
  • Sem linhas — apenas espaço em branco com R:
  • Temas: listar e corrigir os problemas do GerenciadorPedidos, refatorar o método buscar de O(n²) para O(n), escrever dois testes no padrão AAA para o método de desconto, documentar um método com Javadoc completo, descrever o fluxo integrado de 8 passos com as ferramentas de cada etapa
Gabarito
  • Em página separada — uso exclusivo do professor
  • Objetivas: letra correta + justificativa completa
  • Discursivas: resposta esperada com critérios de correção e exemplos de código válidos
