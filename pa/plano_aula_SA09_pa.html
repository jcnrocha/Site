Três Pilares Avançados da Orientação a Objetos

🧬
Herança
"A subclasse herda tudo que a superclasse tem de public/protected — atributos e métodos — e pode especializar ou adicionar comportamentos sem repetir código"
  • Sintaxe: class Cachorro extends Animal { ... }
  • super() → chama o construtor da superclasse (deve ser a 1ª instrução)
  • super.metodo() → reutiliza o método da classe pai dentro da subclasse
  • Java permite herança simples — apenas uma superclasse direta
  • Toda classe em Java herda implicitamente de Object
🎭
Polimorfismo
"Uma referência do tipo pai pode apontar para um objeto filho — e o método chamado é sempre o do tipo real do objeto, não o da referência (ligação dinâmica)"
  • Animal a = new Cachorro(); → referência pai, objeto filho
  • a.fazerSom(); → chama o método de Cachorro, não de Animal
  • @Override anota que o método substitui o da superclasse
  • instanceof → verifica o tipo real do objeto em tempo de execução
  • Permite tratar coleções de tipos diferentes de forma uniforme
📋
Interfaces e Abstração
"Interface = contrato de comportamento — define O QUÊ, não o COMO. Classe abstrata = superclasse parcialmente implementada que força subclasses a completar o que falta"
  • interface Voavel { void voar(); } → assinatura sem corpo
  • implements → classe assina o contrato da interface
  • Uma classe pode implementar múltiplas interfaces
  • abstract class → pode ter métodos implementados e abstratos
  • Método abstract obriga subclasse a implementar — sem corpo na superclasse

Palavras-chave essenciais de herança e polimorfismo em Java

🧬
extends
Estabelece herança — subclasse herda atributos e métodos public/protected da superclasse
🔄
@Override
Anota a sobrescrita de método — o compilador verifica se realmente existe na superclasse
👆
super
super() chama construtor pai · super.met() reutiliza método da superclasse
📋
interface
Contrato de comportamento — define assinaturas de métodos que a classe implementadora deve fornecer
🏛️
abstract
Classe ou método incompleto — não pode ser instanciada, obriga subclasses a implementar os métodos abstratos

Herança ("é-um") × Composição ("tem-um")

🧬
Herança — relação "é-um"
  • Cachorro é-um Animal → faz sentido herdar de Animal
  • Subclasse obtém toda a estrutura da superclasse automaticamente
  • Permite polimorfismo: tratar Cachorro como Animal onde necessário
  • Cuidado: cria acoplamento forte — mudança na superclasse afeta todas as subclasses
  • Regra: use herança somente quando a relação "é-um" for verdadeira e duradoura
  • Java: apenas uma superclasse direta (extends)
Teste: "Cachorro é-um Animal?" — SIM → herança é adequada
🔧
Composição — relação "tem-um"
  • Carro tem-um Motor → Motor não herda de Carro; é um atributo
  • Classe usa outra classe como atributo interno
  • Acoplamento mais fraco — Motor pode ser trocado sem alterar Carro
  • Favorece reuso sem criar hierarquias profundas e frágeis
  • Princípio "Favoritar composição sobre herança" — design patterns modernos
  • Java: sem restrição de número de objetos compostos
Teste: "Carro é-um Motor?" — NÃO → use composição

🎮 Dinâmica É-um ou Tem-um?
🧬 É-um ou Tem-um? — Herança ou Composição
⏱️ Duração: 15 minutos
👥 Agrupamento: grupos de 4 a 5 alunos
🎯 Objetivo: desenvolver o raciocínio de design OO — a escolha entre herança e composição é uma das decisões mais importantes no projeto de software e impacta diretamente a manutenibilidade do código
1
Distribuição dos pares (1 min) — Cada grupo recebe 8 fichas com pares de entidades. Para cada par: decidir se é herança ou composição, aplicar o teste "é-um" vs "tem-um", escrever a assinatura Java correta (extends ou atributo) e justificar em uma frase
2
Análise em grupo (8 min) — O grupo debate e registra. Grupos que terminam antes recebem desafio extra: o par "Gerente × Funcionário" — é herança ou composição? Professor circula, observa e anota respostas divergentes para debate
3
Debate e gabarito (6 min) — Professor projeta cada par, pede a resposta de um grupo e expõe o raciocínio correto; para pares ambíguos, mostra como o contexto do sistema muda a resposta — reforça que design OO tem nuances e que o teste "é-um" é o critério prático
Par de EntidadesRelaçãoAssinatura Java
Cachorro × Animal Herança (é-um) class Cachorro extends Animal
Carro × Motor Composição (tem-um) class Carro { private Motor motor; }
ContaCorrente × Conta Herança (é-uma) class ContaCorrente extends Conta
Pedido × ItemPedido Composição (tem-vários) class Pedido { List<ItemPedido> itens; }
Pássaro × Voavel Interface (implementa) class Passaro extends Animal implements Voavel
Quadrado × Forma Herança (é-uma) class Quadrado extends Forma
Computador × CPU Composição (tem-uma) class Computador { private CPU processador; }
Gerente × Funcionário Herança (é-um) class Gerente extends Funcionario · contexto comum em RH
💡 Reforço pedagógico: o erro mais frequente dos iniciantes em OO é usar herança onde a composição seria mais adequada — criando hierarquias frágeis que se quebram quando os requisitos mudam. O teste "é-um" é simples, mas poderoso: "ContaCorrente é-uma Conta?" — SIM. "Carro é-um Motor?" — NÃO. Se a frase soar estranha, use composição.
