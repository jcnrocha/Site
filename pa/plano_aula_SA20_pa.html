O Padrão AAA — Arrange, Act, Assert

🧰
Arrange — Preparar
"Define os dados de entrada e configura o cenário do teste — tudo o que o método precisa para ser executado"
  • Criar o objeto a ser testado: Calculadora calc = new Calculadora();
  • Definir as entradas: double nota1 = 8.0; double nota2 = 6.0;
  • Cenário isolado: sem banco, rede ou arquivos externos
▶️
Act — Agir
"Chama o método que está sendo testado — uma única ação clara por teste"
  • double resultado = calc.calcularMedia(nota1, nota2);
  • O teste verifica UMA unidade — geralmente um único método
  • Guardar o resultado para a verificação
✅
Assert — Verificar
"Confirma que o resultado obtido é igual ao resultado esperado — é aqui que o teste passa ou falha"
  • Java: assertEquals(7.0, resultado);
  • Python: assert resultado == 7.0
  • Verde: passou · Vermelho: falhou — a mensagem mostra esperado × obtido

Cinco Grupos de Assertions Mais Usadas

⚖️
assertEquals
Valor obtido é igual ao esperado · assertNotEquals verifica o contrário
✔️
assertTrue / False
A condição é verdadeira ou falsa — ideal para métodos booleanos
⭕
assertNull / NotNull
O objeto é nulo ou não nulo — verifica criação e retorno de objetos
💥
assertThrows
O código lança a exceção esperada — testa cenários de erro
🏷️
Nomenclatura
Padrão deve[Comportamento]Quando[Condição] · classe CalculadoraTest na pasta test

Teste Manual × Teste Automatizado

👁️
Teste Manual
  • Velocidade: lento — rodar o programa e verificar visualmente
  • Consistência: depende do testador — cada execução pode ser diferente
  • Reexecução: trabalhosa — refazer tudo a cada alteração
  • Documentação: nenhuma
  • Cobertura: limitada ao que foi verificado naquele momento
Não escala — inviável conforme o sistema cresce
🤖
Teste Automatizado
  • Velocidade: milissegundos — código que verifica código
  • Consistência: sempre igual — mesma verificação toda vez
  • Reexecução: um clique — a qualquer momento
  • Documentação: o próprio código de teste descreve o comportamento
  • Cobertura: todos os cenários escritos, sempre verificados
Garante que nada quebrou após qualquer alteração
✅ Executando na IDE: VS Code com extensão de teste exibe um botão de play ao lado de cada método de teste · resultado verde = passou · vermelho = falhou, com a mensagem indicando valor esperado × valor obtido — o que facilita a correção. Teste unitário é isolado (sem banco, rede ou arquivos), automatizado e rápido — cada teste roda em milissegundos.

🎮 Dinâmica Escreve o Teste
🧪 Escreve o Teste — Padrão AAA na Prática
⏱️ Duração: 15 minutos
👥 Agrupamento: grupos de 4 a 5 alunos
🎯 Objetivo: aplicar o padrão AAA escrevendo testes unitários para métodos simples — desenvolvendo a habilidade de pensar nos cenários de teste antes mesmo de escrever o código de produção, prática conhecida como TDD (Test-Driven Development)
1
Apresentação do método (1 min por rodada) — O professor projeta a assinatura e o comportamento esperado de um método. O grupo deve escrever o teste unitário usando o padrão AAA — sem ver a implementação do método
2
Escrita do teste em grupo (8 min) — Os grupos escrevem os testes na ficha ou diretamente na IDE, cobrindo ao menos dois cenários: o cenário feliz (entrada válida) e o cenário de borda (entrada no limite ou inválida)
3
Execução e debate (6 min) — O professor apresenta a implementação dos métodos · os grupos executam seus testes · verificam se passam ou falham · discutem os cenários que não foram cobertos
MétodoComportamento esperadoCenários sugeridos
boolean ehMaiorDeIdade(int idade) Retorna true se idade >= 18 idade = 18 (limite) · idade = 17 · idade = 0
double calcularDesconto(double preco, double percentual) Retorna preço com desconto aplicado preco=100, percentual=10 → 90.0 · percentual=0 → preço original
String classificarNota(double nota) Retorna "Aprovado", "Recuperação" ou "Reprovado" nota=7.0 · nota=5.0 · nota=4.9
boolean cpfValido(String cpf) Retorna true se CPF tem 11 dígitos numéricos CPF válido · CPF com letras · CPF com menos dígitos
💡 Reforço pedagógico: escrever o teste antes de ver a implementação é uma prática avançada chamada TDD — Test-Driven Development. O teste funciona como uma especificação executável: descreve exatamente o que o método deve fazer em cada situação. Empresas que adotam TDD têm menos bugs em produção e código mais bem estruturado.
