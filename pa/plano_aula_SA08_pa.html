Três Fundamentos da Orientação a Objetos

🏗️
Classes e Objetos
"A classe é o molde — define o que o objeto terá. O objeto é a instância — ocupa memória e tem seu próprio estado em tempo de execução"
  • Classe: class ContaBancaria { ... }
  • Objeto: ContaBancaria c = new ContaBancaria();
  • Cada objeto tem seus próprios valores de atributo (estado independente)
  • Dois objetos da mesma classe não compartilham estado
  • Referência nula: ContaBancaria c = null; → NullPointerException se usada
📊
Atributos e Métodos
"Atributos guardam o estado do objeto; métodos definem seu comportamento — juntos representam o que o objeto É e o que ele PODE FAZER"
  • Atributo: variável de instância declarada na classe (double saldo;)
  • Método: função que opera sobre os atributos do próprio objeto
  • Acesso via ponto: conta.depositar(500);
  • this → referência ao próprio objeto dentro de um método
  • Construtor: método especial com o nome da classe, chamado no new
🔒
Encapsulamento
"Proteger os dados internos do objeto — ninguém altera o saldo diretamente; a mudança só acontece via métodos que validam a operação"
  • private: acessível apenas dentro da própria classe
  • public: acessível de qualquer lugar
  • Getter: public double getSaldo() { return saldo; }
  • Setter com validação: if(valor > 0) this.saldo += valor;
  • Objetivo: invariantes de negócio garantidas na classe, não no chamador

Os 4 Pilares da OO + Classe (visão geral)

🏗️
Classe / Objeto
Classe = molde · Objeto = instância com estado próprio criada com new
🔒
Encapsulamento
Atributos private protegidos por getters/setters com validação de negócio
🧬
Herança
Classe filha reutiliza e especializa a classe pai com extends · Aprofundado na SA09
🎭
Polimorfismo
Mesmo método, comportamentos diferentes conforme o tipo real do objeto · Aprofundado na SA09
🗂️
Abstração
Expor apenas o essencial e ocultar a implementação — o chamador sabe O QUE o método faz, não COMO

Paradigma OO × Paradigma Estruturado

🏗️
Orientado a Objetos (OO)
  • Código organizado em classes que modelam entidades reais (Conta, Produto, Aluno)
  • Dados e comportamentos juntos no mesmo objeto — coesão natural
  • Encapsulamento protege dados de alterações indevidas
  • Herança permite reutilizar e especializar comportamentos sem duplicar código
  • Polimorfismo permite tratar diferentes tipos de forma uniforme
  • Ideal para sistemas com muitos tipos de entidade e regras de negócio complexas
Use quando: modelar entidades do mundo real com estado e comportamento próprios
⚙️
Estruturado (Procedural)
  • Código organizado em funções que operam sobre dados separados
  • Dados e funções separados — estruturas passadas como parâmetro
  • Sem encapsulamento nativo — dados podem ser acessados diretamente
  • Reutilização via funções e módulos — sem herança ou polimorfismo
  • Mais simples para problemas lineares e sequenciais
  • Ideal para scripts, algoritmos matemáticos e processamento de dados simples
Use quando: problema é sequencial, sem entidades complexas com estado próprio

💡 Reforço pedagógico: modelar antes de codificar é o hábito que separa o desenvolvedor iniciante do profissional. Pensar em atributos e métodos é pensar na responsabilidade de cada entidade — o mesmo princípio da responsabilidade única da SA07, agora aplicado a objetos. Regra prática: se você não consegue nomear o método em um verbo claro, a responsabilidade ainda não está bem definida.
