As Três Frentes da Otimização

⏱️
Medir Primeiro
"Não se otimiza o que não se mede — a intuição engana, o cronômetro não"
  • Java: long inicio = System.currentTimeMillis(); antes e depois do trecho
  • Python: inicio = time.time() · diferença em segundos
  • Comparar o tempo ANTES e DEPOIS da otimização — só assim se comprova o ganho
  • Otimizar onde o programa realmente passa tempo, não onde "parece" lento
🔁
Otimizar Laços
"O que está dentro de um laço executa milhares de vezes — todo desperdício ali é multiplicado"
  • Mover cálculos que não mudam para FORA do laço
  • Concatenação de String com + em laço cria um novo objeto a cada volta — usar StringBuilder
  • Evitar laços aninhados desnecessários — cada nível multiplica as iterações
  • Interromper a busca com break quando o item já foi encontrado
🗄️
Otimizar Acesso a Dados
"A operação mais lenta do sistema é a ida ao banco de dados — reduzir o número de idas é a maior otimização"
  • NUNCA fazer SELECT dentro de um laço — buscar tudo de uma vez com WHERE ou IN
  • Reaproveitar a conexão aberta em vez de abrir e fechar a cada operação
  • Guardar em variável o resultado que será usado várias vezes (cache simples)
  • Selecionar apenas as colunas necessárias em vez de SELECT *

Cinco Técnicas de Otimização Trabalhadas em Aula

🧵
StringBuilder
Para concatenar texto em laço: sb.append(...) em vez de str += ...
📤
Fora do Laço
Cálculos e chamadas que não mudam a cada iteração são feitos uma única vez, antes do laço
🛑
break na Busca
Encontrou o item? Sair do laço — continuar percorrendo o resto é desperdício
🗄️
Consulta Única
Uma consulta com WHERE/IN no lugar de N consultas dentro de um laço
💾
Cache Simples
Guardar em variável o resultado caro que será reutilizado — evita recalcular ou rebuscar

Otimização Prematura × Otimização Consciente

🏃
Otimização Prematura — o erro
  • Otimizar antes de medir — "achar" que algo é lento
  • Sacrificar a legibilidade por microganhos irrelevantes
  • Código "esperto" que ninguém da equipe entende depois
  • Tempo gasto otimizando trecho que executa uma vez por dia
  • Frase clássica de Donald Knuth: "otimização prematura é a raiz de todo mal"
Sem medição, a "otimização" pode nem ser otimização
🎯
Otimização Consciente — o caminho
  • 1º: escrever código correto e legível
  • 2º: medir — identificar onde o tempo realmente é gasto
  • 3º: otimizar o gargalo comprovado, preservando a clareza
  • 4º: medir de novo — confirmar o ganho com números
  • Priorizar gargalos que afetam o usuário: laços grandes, banco, rede
Medir → otimizar → medir de novo — sempre nessa ordem

Gargalos Comuns — Antes e Depois

Código lento (antes)GargaloCódigo otimizado (depois)
String s = ""; for (...) { s += item; } Novo objeto String a cada volta StringBuilder sb; for (...) { sb.append(item); }
for (...) { double taxa = calcularTaxa(); ... } Cálculo repetido sem necessidade double taxa = calcularTaxa(); for (...) { ... }
SELECT executado dentro de um laço de 500 clientes 500 idas ao banco Uma única consulta com WHERE id IN (...) antes do laço
Busca que percorre o vetor inteiro mesmo após encontrar Iterações desperdiçadas if (encontrado) { break; } ao achar o item
⚖️ Equilíbrio legibilidade × desempenho: código otimizado mas ilegível custa caro em manutenção. A regra prática da aula: otimize o que foi medido e comprovado como gargalo, mantenha o restante o mais simples e claro possível — um código limpo é mais fácil de otimizar depois do que um código "esperto" é de entender.

🚀 Acelera o Código — Caça aos Gargalos
⏱️ Duração: 15 minutos
👥 Agrupamento: grupos de 4 a 5 alunos
🎯 Objetivo: identificar gargalos de desempenho em trechos de código reais e propor a otimização adequada — desenvolvendo o raciocínio de análise de desempenho que o desenvolvedor usa quando um sistema fica lento
1
Apresentação dos trechos (2 min) — O professor projeta 4 trechos de código funcionais mas lentos. Para cada um, o grupo deve: identificar o gargalo · explicar por que ele torna o código lento · propor a versão otimizada
2
Análise em grupo (8 min) — Os grupos registram na ficha o gargalo de cada trecho e escrevem a estrutura da versão otimizada. Grupos que terminarem antes recebem o desafio extra: ordenar os 4 gargalos do maior para o menor impacto
3
Correção com medição ao vivo (5 min) — O professor revela cada gargalo e executa as versões antes/depois na IDE exibindo os tempos — os números comprovam (ou surpreendem) as estimativas dos grupos
Trecho lentoGargaloOtimização proposta
Relatório montado com String += dentro de laço de 100.000 linhas Concatenação em laço Substituir por StringBuilder.append()
Math.pow(taxa, 2) recalculado em cada uma das 50.000 iterações Cálculo invariante no laço Calcular uma vez antes do laço e guardar em variável
SELECT do nome do cliente executado dentro do laço de pedidos Banco dentro de laço Uma única consulta antes do laço com WHERE id IN (...)
Busca de um aluno que continua percorrendo o vetor após encontrá-lo Iterações desnecessárias Adicionar break assim que o aluno for encontrado
💡 Reforço pedagógico: no mercado, problemas de desempenho quase sempre aparecem quando o sistema cresce — o código que funcionava com 100 registros trava com 100.000. Desenvolvedores que sabem medir e identificar gargalos resolvem em minutos o que outros levam dias "chutando". E a lição mais importante: sem medição não há otimização — há suposição.
