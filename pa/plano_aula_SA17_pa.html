Três Formas de Reutilizar Código

🔁
Métodos Reutilizáveis
"Método que faz uma coisa bem feita é reutilizável — método que faz muitas coisas só serve para aquele contexto"
  • Uma responsabilidade por método
  • Parâmetros em vez de valores fixos (hardcoded)
  • Retorno em vez de exibição direta no console
  • Ex.: boolean ehMaiorDeIdade(int idade) > void verificarIdade()
📦
Bibliotecas Externas
"Código desenvolvido, testado e mantido pela comunidade — evita reinventar a roda"
  • Gerenciadas por Maven/Gradle (Java), pip (Python), npm (JS)
  • Datas: java.time · HTTP: requests
  • Validação: commons-validator · JSON: jackson
  • PDF: reportlab via pip install
🧰
Classes Utilitárias e OO
"Classe utilitária agrupa métodos estáticos reutilizáveis — e a OO bem aplicada é DRY na estrutura do sistema"
  • ValidadorUtils.emailValido(email) usável em todo o sistema
  • Não representa entidade do negócio — é uma coleção de ferramentas
  • Herança reutiliza estrutura da classe pai nas filhas
  • Interfaces definem contratos reutilizados por classes diferentes

Cinco Conceitos-Chave da Reutilização

🌵
DRY
Don't Repeat Yourself — cada lógica com uma única representação no sistema
💦
WET
Write Everything Twice (ou Waste Everyone's Time) — o oposto do DRY
🎯
Uma Responsabilidade
Método que faz uma coisa bem feita — reutilizável em qualquer contexto
🔧
Parâmetros
Substituem valores fixos e tornam o método genérico para vários cenários
↩️
Retorno
Método que retorna valor é mais reutilizável que o que apenas exibe no console

Código WET × Código DRY

💦
WET — Código com Repetição
  • Mesma lógica de média e aprovação copiada para cada aluno
  • mediaAluno1, mediaAluno2, mediaAluno3... — variações da mesma coisa
  • Se a regra mudar, é preciso alterar em todos os lugares
  • Esquecer uma cópia gera comportamentos inconsistentes — bugs difíceis de encontrar
  • Aumenta o tamanho do código sem agregar valor
Sinal de alerta: copiar e colar lógica com pequenas variações
🌵
DRY — Lógica Centralizada
  • calcularMedia(n1, n2, n3) — cálculo em um único lugar
  • verificarAprovacao(nome, media) — decisão de negócio centralizada
  • Uso: verificarAprovacao("Aluno 1", calcularMedia(a, b, c))
  • Regra mudou? Uma alteração, em um único lugar
  • Centraliza decisões de negócio — não é só evitar copiar e colar
Extrair a lógica repetida para método, função ou classe reutilizável

Sinais de que o Código Precisa Ser Refatorado para DRY

SinalO que indica
Trecho duplicado Mesmo código aparece em dois ou mais lugares com pequenas variações
Bug corrigido em vários arquivos Ao corrigir um bug, é necessário alterar em mais de um arquivo — a lógica está espalhada
Método com mais de 30 linhas Provavelmente está fazendo mais de uma coisa — dividir em métodos menores
Classe com mais de 300 linhas Provavelmente acumula responsabilidades demais — extrair classes utilitárias ou de domínio

🌵 Seca o Código — Eliminando Repetições com DRY
⏱️ Duração: 15 minutos
👥 Agrupamento: grupos de 4 a 5 alunos
🎯 Objetivo: aplicar o princípio DRY identificando e eliminando repetições em trechos de código reais — desenvolvendo o raciocínio de refatoração que os desenvolvedores usam continuamente para manter a qualidade do código ao longo do tempo
1
Apresentação do código (2 min) — O professor projeta um trecho com repetição evidente: três relatórios de aluno com a mesma estrutura de cálculo de média e situação. Os grupos precisam: identificar o trecho repetido · extrair para um método reutilizável · reescrever o código principal usando o novo método
2
Refatoração em grupo (8 min) — Os grupos reescrevem o código aplicando DRY na ficha de refatoração
3
Comparação e debate (5 min) — Os grupos apresentam suas versões. O professor compara e discute: "Qual versão é mais fácil de manter? O que acontece se a regra de negócio mudar?"
Código WET (repetido 3×)Extração DRYUso final
Cálculo (n1 + n2 + n3) / 3 copiado para Ana, Carlos e Julia Método de cálculo double calcularMedia(n1, n2, n3) gerarRelatorio("Ana", 8.5, 7.0, 9.0);
gerarRelatorio("Carlos", 5.0, 4.5, 6.0);
gerarRelatorio("Julia", 9.5, 8.0, 10.0);
Bloco if (media >= 7.0) ... else ... repetido em cada relatório Decisão centralizada dentro de gerarRelatorio(...)
Cabeçalho e separador "=== Relatório ===" impressos 3 vezes Impressão única no método gerarRelatorio(nome, n1, n2, n3)
💡 Reforço pedagógico: se a nota mínima para aprovação mudar de 7.0 para 6.5, na versão WET seriam necessárias 3 alterações — e esquecer uma causaria inconsistência. Na versão DRY: 1 alteração em um único lugar. Isso é o que DRY significa na prática.
