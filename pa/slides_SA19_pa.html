O primeiro passo para corrigir é classificar

🐞 Os Três Tipos de Erro

Tipo 1 — Mais Fácil Erro de Sintaxe
O código não compila. A IDE sublinha em vermelho — falta de ;, chave não fechada, variável não declarada, parêntese faltando.

Estratégia: siga o indicador da IDE — ela aponta a linha exata do problema.
Tipo 2 — Mais Traiçoeiro Erro de Lógica
O código executa, mas o resultado está errado. Laço infinito, cálculo com operador errado, condição invertida, divisão por 2 em vez de 3.

Estratégia: depurador — inspecionar variáveis passo a passo até achar onde o valor diverge do esperado.
Tipo 3 — Para Abruptamente Erro de Runtime
O programa para com uma exceção. NullPointerException, ArrayIndexOutOfBoundsException, divisão por zero, ClassCastException.

Estratégia: ler o stack trace — ele diz a linha exata. Breakpoint nela, verificar qual referência é null.

Regra de ouro: sempre leia a mensagem de erro completa antes de mexer no código. O stack trace diz onde, o tipo da exceção diz o quê — juntos eles apontam para o problema sem precisar adivinhar.

Dois jeitos de depurar — resultados muito diferentes

🖨️ println × 🔬 Depurador

🖨️ println — o jeito iniciante
✗ Modifica o código só para inspecionar valores
✗ Recompila e executa de novo a cada nova suspeita
✗ Deixa "lixo" no código — prints esquecidos chegam a produção
✗ Mostra apenas o que foi explicitamente impresso
✗ Ineficaz em laços longos — imprime 10.000 linhas
✗ Lento: cada suspeita é uma rodada de recompilação
🔬 Depurador — o jeito profissional
✔ Inspeciona qualquer variável sem modificar uma linha
✔ Pausa exatamente onde interessa — inclusive com condição
✔ Navega o fluxo passo a passo: step over, into, out
✔ Painel Variables mostra todo o escopo atual de uma vez
✔ Breakpoint condicional pausa só na iteração problemática
✔ Call Stack revela o caminho completo até o erro

No mercado, desenvolvedores que sabem usar o depurador encontram bugs em minutos — o que levaria horas de prints. É uma das habilidades mais valorizadas em entrevistas e no dia a dia de qualquer equipe de desenvolvimento.

O kit de ferramentas do depurador

🔬 Cinco Ferramentas Essenciais do Depurador

🔴
Breakpoint — Clique na margem esquerda da linha desejada. A execução pausa exatamente nela. Breakpoint condicional: clique direito → "Edit Breakpoint" → coloque a condição (ex: i == 537) — pausa APENAS naquela iteração.
F10
Step Over — Executa a linha atual e avança para a próxima, sem entrar dentro de métodos chamados. Use quando você não precisa inspecionar o interior de uma função que já funciona.
F11
Step Into — Entra dentro do método chamado na linha atual, permitindo inspecionar sua execução. Use quando o bug pode estar dentro da função chamada.
⇧F11
Step Out — Termina a execução do método atual e retorna ao código que o chamou. Use quando entrou em um método e quer sair rapidamente sem step over linha a linha.
👁️
Variables / Watch / Call Stack — Variables: todos os valores no escopo atual. Watch: monitore uma expressão customizada. Call Stack: a pilha de chamadas — qual método chamou qual até o ponto pausado. Hover sobre qualquer variável no código exibe seu valor.
Onde colocar e como usar

🔴 Breakpoints na Prática

Cenário: NullPointerException
// Erro: NullPointerException na linha 8
public void processarPedidos(List<Pedido> pedidos) {
  for (Pedido p : pedidos) {
    // ← BREAKPOINT AQUI (clique na margem)
    String nome = p.getCliente().getNome();
    System.out.println(nome);
  }
}
Com o breakpoint na linha 4, o depurador pausa em cada iteração. O painel Variables mostra: p → inspecionar se p.getCliente() é null antes de chamar getNome().
Breakpoint Condicional — o poder real
for (int i = 0; i < 10_000; i++) {
  processar(pedidos[i]);
}                         
// O erro acontece apenas em i = 537
// Breakpoint simples: F10 × 537 = inviável
// Breakpoint condicional: i == 537
// → pausa SOMENTE na iteração problemática
Como criar: clique direito no ponto vermelho do breakpoint → Edit Breakpoint → digite a condição. O depurador ignora todas as iterações até que a condição seja verdadeira.
Navegar e inspecionar — os dois movimentos do depurador

⏭️ Navegar Passo a Passo

Comandos de Navegação
// Situação: pausado na linha → calcularMedia(notas)

F10 Step Over
  → executa a chamada e avança
  → não entra dentro de calcularMedia

F11 Step Into
  → entra DENTRO de calcularMedia
  → inspeciona linha a linha do método

Shift+F11 Step Out
  → sai de calcularMedia e retorna

F5 Continue
  → continua até o próximo breakpoint
Painéis de Inspeção
Variables
Todos os valores no escopo atual — variáveis locais, parâmetros e campos do objeto. Expanda objetos para ver seus atributos.
Watch
Monitore uma expressão customizada que a IDE avalia automaticamente a cada passo. Exemplo: lista.size() ou nota >= 7.0.
Call Stack
A pilha de chamadas — qual método chamou qual, de baixo para cima. Essencial para entender como a execução chegou até o ponto pausado.
O depurador do ambiente de produção

📜 Logs e Rastreabilidade

Em produção, o depurador não existe. Logs são o único mecanismo de rastreabilidade de um programa em execução real — registrar os eventos certos, no nível certo, é o que permite diagnosticar um incidente sem reproduzi-lo localmente.

☕ Java — java.util.logging
import java.util.logging.Logger;
private static final Logger log =
  Logger.getLogger(SistemaAluno.class.getName());

log.fine("DEBUG: nota recebida = " + nota);
log.info("Aluno " + nome + " cadastrado");
log.warning("Login inválido: " + usuario);
log.severe("Falha na conexão com banco!");
🐍 Python — módulo logging
import logging
logging.basicConfig(level=logging.DEBUG)

logging.debug(f"nota recebida: {nota}")
logging.info(f"aluno {nome} cadastrado")
logging.warning(f"login inválido: {usuario}")
logging.error("falha na conexão com banco!")

DEBUG — valores internos, só em desenvolvimento · INFO — eventos normais (login, pedido criado) · WARN — situação inesperada mas não crítica · ERROR — falha que impede uma operação. Em produção, filtre para INFO ou acima — DEBUG inunda o arquivo de log.

O método que profissionais usam

🎯 Estratégia de Depuração em 6 Passos

01
Reproduzir — Fazer o erro acontecer de forma consistente e controlada. Sem reprodução confiável não há depuração — é jogo de sorte. Anote os passos exatos que causam o problema.
02
Isolar — Identificar o menor trecho de código que ainda reproduz o erro. Quanto menor, mais fácil de analisar. Elimine partes até não poder mais sem que o bug desapareça.
03
Breakpoint — Posicionar o breakpoint antes de onde o erro acontece — nunca depois. Condicional se for em laço com muitas iterações. Use Call Stack para escolher o ponto mais alto na pilha.
04
Inspecionar — Executar passo a passo e observar as variáveis no painel. Verificar em que momento o valor diverge do esperado — essa é a causa raiz, não o sintoma visível.
05
Corrigir — Aplicar a correção identificada. Não corrija vários problemas de uma vez — uma mudança por vez, para saber o que resolveu.
06
Verificar — Confirmar que o erro não ocorre mais e que a correção não introduziu novos problemas. Execute o fluxo completo, não apenas o caso que estava bugado.
Os bugs mais frequentes e como rastreá-los

📋 Erros Comuns — Sintoma e Estratégia

Erro Sintoma Tipo Estratégia de depuração
NullPointerException Programa para com NPE, stack trace indica a linha Runtime Breakpoint na linha indicada → Variables → qual referência é null → verificar se objeto foi inicializado
Resultado errado em cálculo Valor exibido diferente do esperado Lógica Breakpoint antes do cálculo → Step Over em cada operação → Watch para cada variável → achar onde o valor diverge
Laço infinito Programa trava sem resposta Lógica Pausar execução manualmente → Variables → verificar condição de saída do laço → identificar qual variável não avança
Dado não salvo no banco Operação parece funcionar, dado não persiste Lógica Verificar se commit() está sendo chamado → Log INFO após cada operação → verificar se exception silenciosa engole o erro
ArrayIndexOutOfBounds Exceção na última iteração de um for Runtime Verificar condição do for — geralmente i <= array.length em vez de i < array.length

Padrão recorrente: leia o stack trace primeiro. Ele indica o arquivo, linha e tipo de exceção — 90% do diagnóstico já está ali antes de abrir o depurador.

Atividade prática em grupo

🐞 Dinâmica — Caçada ao Bug

🔬
Depuração de Programas com Bugs Intencionais · 15 minutos · grupos de 4 a 5 alunos
01
Distribuição dos programas (1 min) — Cada grupo recebe um programa diferente com um bug intencional. O desafio: identificar o tipo de erro, usar o depurador para encontrar a causa raiz, corrigir e descrever o processo usado.
02
Depuração em grupo (10 min) — Os grupos usam o depurador — breakpoints, step over/into e painel de variáveis — para encontrar e corrigir o bug. Registram na ficha: onde colocaram o breakpoint · o que observaram no painel Variables · qual era o erro · como corrigiram.
03
Relato e debate (4 min) — Cada grupo relata o bug encontrado e o processo de depuração. O professor complementa com dicas adicionais de cada caso e exibe ao vivo.

Os 4 bugs: calculadora de média dividindo por 2 em vez de 3 · busca com i <= lista.length causando ArrayIndexOutOfBounds · commit() comentado · verificação de senha com = em vez de ==.

O que vimos hoje

📌 Resumo da Aula

🐞 Três Tipos de Erro
Sintaxe — não compila, IDE sublinha · Lógica — executa mas resultado errado, o mais difícil, precisa do depurador · Runtime — para abruptamente com exceção, leia o stack trace primeiro.
🔬 Ferramentas do Depurador
F9 adiciona breakpoint · F5 inicia debug e continua · F10 step over · F11 step into · Shift+F11 step out. Painéis: Variables, Watch e Call Stack — o escopo completo sem nenhum print.
📜 Logs com Níveis
DEBUG — fluxo interno, só em dev · INFO — eventos normais · WARN — situação inesperada não crítica · ERROR — falha que impede operação. Em produção, logs substituem o depurador.
🎯 Estratégia em 6 Passos
Reproduzir → Isolar → Breakpoint → Inspecionar → Corrigir → Verificar. Nunca corrija sem reproduzir, nunca feche sem verificar. Uma mudança por vez — para saber o que resolveu.

A lição mais importante: depurar é uma habilidade, não um talento. Quanto mais você praticar com o depurador, mais rápido ficará — e no mercado, isso vale ouro.
