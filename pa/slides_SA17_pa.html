Por que copiar e colar é perigoso

💦 WET × DRY — O Princípio da Reutilização

DRY — Don't Repeat Yourself: cada lógica tem uma única representação no sistema · WET — Write Everything Twice: mesma lógica copiada em vários lugares — o oposto do DRY

💦 CÓDIGO WET — PERIGO Mesma lógica em vários lugares
Cálculo de média copiado para Ana, Carlos e Julia · se a regra mudar, precisa alterar em todos os lugares · esquecer uma cópia causa comportamentos inconsistentes — bugs difíceis de encontrar · código cresce sem agregar valor
🌵 CÓDIGO DRY — CERTO Lógica centralizada em um único lugar
calcularMedia(n1, n2, n3) — um único lugar · se a regra mudar, alterar em um único lugar · código menor, mais claro e mais fácil de manter · DRY centraliza decisões de negócio, não é só "não copiar e colar"

Sinal de alerta: se você está copiando e colando lógica com pequenas variações, o código precisa de um método reutilizável com parâmetros

Comparando os dois cenários

⚖️ Código WET × Código DRY

💦 WET — O que evitar
✗ Cálculo de média copiado para cada aluno
✗ Bloco if/else de aprovação repetido em cada relatório
✗ Mudar a regra exige alterar em vários arquivos
✗ Esquecer uma cópia gera inconsistência e bugs
✗ Código cresce em tamanho sem agregar valor real
🌵 DRY — O que fazer
✔ calcularMedia(n1, n2, n3) — cálculo em um único lugar
✔ gerarRelatorio(nome, n1, n2, n3) — decisão centralizada
✔ Regra mudou? Uma alteração, em um único lugar
✔ Uso: gerarRelatorio("Ana", 8.5, 7.0, 9.0)
✔ Centraliza decisões de negócio no sistema inteiro

Teste do DRY: se ao alterar uma regra de negócio você precisa mexer em mais de um lugar — o código está WET e precisa de refatoração

Refatoração — do código repetido ao reutilizável

💻 Código WET × Código DRY

O código WET tem a mesma estrutura copiada 3 vezes — para Ana, Carlos e Julia. O DRY extrai tudo para um único método com parâmetros

✗ WET — repetido 3 vezes
double m1 = (8.5+7.0+9.0)/3; if (m1 >= 7.0) { System.out.println("Ana: Aprovado"); } else { System.out.println("Ana: Reprovado"); } // ↑ mesmo padrão copiado para // Carlos e Julia...
✔ DRY — um único método
static void gerarRelatorio( String nome, double n1, double n2, double n3) { double media = (n1+n2+n3)/3; String sit = media >= 7.0 ? "Aprovado" : "Reprovado"; System.out.println(nome+": "+sit); } // gerarRelatorio("Ana",8.5,7.0,9.0);

Versão WET: 3 alterações para mudar a nota mínima de 7.0 para 6.5 · Versão DRY: 1 alteração em um único lugar

Uma responsabilidade, parâmetros e retorno

🔁 Método Reutilizável — Como Construir

Método reutilizável tem uma responsabilidade, recebe parâmetros em vez de valores fixos e retorna resultado em vez de imprimir diretamente — pode ser chamado em qualquer contexto

✗ NÃO REUTILIZÁVEL Valor fixo e impressão direta
void verificarIdade() { int idade = 17; if (idade >= 18) ... } — a lógica está presa a um único contexto e não pode ser reaproveitada em outro lugar do sistema
✔ REUTILIZÁVEL Parâmetro e retorno
boolean ehMaiorDeIdade(int idade) { return idade >= 18; } — pode ser chamado com qualquer valor, em qualquer parte do sistema, e o resultado pode ser usado em outros cálculos

Regra: método que retorna é mais versátil que método que apenas exibe no console — o chamador decide o que fazer com o resultado

Extraindo lógica repetida para método

📋 Como Extrair um Método Reutilizável

Ao identificar código WET, siga estes passos para refatorar — transformar repetição em reutilização

01
Identificar o trecho repetido — localizar onde a mesma lógica aparece com pequenas variações: cálculos, validações, formatações de saída
02
Definir a responsabilidade — o método deve fazer uma coisa bem feita: calcularMedia() · validarEmail() · formatarCPF() — nome que se explica
03
Definir os parâmetros — substituir os valores fixos por parâmetros: double n1, double n2, double n3 em vez de valores hard-coded no corpo do método
04
Definir o retorno — o método retorna o resultado em vez de imprimir: return media em vez de System.out.println(media)
05
Substituir os trechos repetidos — chamar o novo método em todos os lugares onde o código estava copiado — testar cada chamada
Não reinventar a roda

📦 Bibliotecas Externas — Código Testado pela Comunidade

Bibliotecas externas são conjuntos de código desenvolvidos, testados e mantidos pela comunidade — evitam reinventar soluções para problemas já resolvidos e amplamente validados

JAVA — MAVEN E GRADLE Gerenciadores de dependências
Declarar no pom.xml e o Maven baixa automaticamente · Datas: java.time (nativo) · Validação: commons-validator · JSON: Jackson ou Gson · HTTP: OkHttp · PDF: iText · conexão DB: JDBC
PYTHON — PIP · JS — NPM Ecossistemas de pacotes
Python: requests (HTTP) · reportlab (PDF) · pandas (dados) · pip install nome · JS: axios · lodash · jest · npm install nome · todas com documentação pública e código aberto

Antes de escrever: pesquise se existe uma biblioteca confiável que já resolva o problema — evita bugs em código reinventado e ganha manutenção gratuita da comunidade

Centralizando ferramentas do sistema

🧰 Classes Utilitárias — Ferramentas Compartilhadas

Classe utilitária agrupa métodos estáticos reutilizáveis que não representam entidades do negócio — são ferramentas que qualquer parte do sistema pode usar sem criar uma instância

🧰 CLASSE UTILITÁRIA Coleção de ferramentas estáticas
ValidadorUtils.emailValido(email) · FormatadorUtils.cpfFormatado(cpf) · CalculoUtils.media(n1, n2, n3) · não representa uma entidade do negócio · métodos estáticos — chamados sem criar instância
🔄 HERANÇA E INTERFACES Reutilização estrutural com OO
Herança: Funcionario agrupa atributos comuns de Gerente, Vendedor e Analista · Interface: Calculavel define o contrato calcularSalario() para diferentes tipos · DRY na estrutura do sistema

ValidadorUtils ≠ Aluno: classe utilitária não é entidade do negócio — é uma coleção de ferramentas de uso geral, usada por todo o sistema

Por que retorno é melhor que impressão direta

⚖️ Método que Exibe × Método que Retorna

✗ Método que exibe — menos reutilizável
✗ void calcularMedia() — imprime direto no console
✗ Lógica acoplada à exibição — não pode ser testado facilmente
✗ Resultado não pode ser usado em outro cálculo
✗ Mudar o formato de exibição exige reescrever o método
✗ Não tem valor de retorno para verificar em testes
✔ Método que retorna — reutilizável
✔ double calcularMedia(n1,n2,n3) — retorna o resultado
✔ O chamador decide: exibir, salvar, comparar, calcular
✔ if (calcularMedia(a,b,c) >= 7) — combinável com lógica
✔ Testável: assertEquals(8.0, calcularMedia(7, 9, 8))
✔ Reutilizável em diferentes contextos e formatos de saída

Regra: o método deve calcular e retornar — a camada de apresentação (UI, console, relatório) decide como exibir o resultado
