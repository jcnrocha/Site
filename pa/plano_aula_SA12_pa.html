Revisão Rápida dos Pilares da OO — na Prática

🔒
Encapsulamento
"Atributos private + getters e setters com validação real — a classe protege os próprios dados contra valores inválidos"
  • Atributos sempre private
  • Acesso somente por get/set públicos
  • Setter com validação: nota só aceita 0–10
  • Quem está fora da classe não conhece a regra interna
🧬
Herança
"Classe pai concentra os atributos comuns — classes filhas herdam com extends e especializam com @Override"
  • Atributos comuns sobem para a classe pai
  • extends herda estrutura e comportamento
  • super() chama o construtor do pai
  • @Override especializa o método herdado
🎭
Polimorfismo
"Referência pai apontando para objetos filhos em um array — cada objeto responde com a sua própria implementação"
  • Pessoa[] pessoas = ... guarda Aluno e Professor
  • Laço for chama exibirDados() de cada um
  • O método executado é o do tipo real do objeto
  • Ligação dinâmica em ação — vista ao vivo na IDE

Cinco Erros Comuns de OO — e Como Identificá-los

💥
NullPointer
Exception
Objeto não foi instanciado com new antes do uso — verificar a criação do objeto
🔍
cannot find symbol
Atributo private acessado diretamente de fora da classe — usar o getter correspondente
🔄
@Override incorreto
Falta da anotação ou assinatura diferente — conferir nome, parâmetros e retorno iguais ao método pai
📏
IndexOutOf
Bounds
Acesso a índice inexistente no vetor — usar .length no laço for em vez de número fixo
🏗️
Construtor não encontrado
Chamada de new Classe(params) sem construtor correspondente — criar o construtor correto

Refatoração — de Estruturado para Orientado a Objetos

📜
Antes — Código Estruturado
  • Variáveis soltas: nomeProduto, precoProduto, quantidadeEstoque
  • Funções que recebem todos os dados por parâmetro a cada chamada
  • exibirProduto(nome, preco, qtd) · aplicarDesconto(preco, perc)
  • Dados e comportamentos separados — nada impede valores inválidos
  • Difícil de reutilizar para um segundo produto
Funciona, mas é desorganizado e não escala
🏗️
Depois — Código Orientado a Objetos
  • Classe Produto agrupa dados e comportamentos
  • Atributos private: nome, preco, quantidadeEstoque
  • Métodos sem repassar dados: exibirDados(), aplicarDesconto(perc), atualizarEstoque(qtd)
  • Construtor + getters/setters com validação
  • Criar 10 produtos = instanciar 10 objetos — reuso imediato
Código mais limpo, organizado e reutilizável

🎮 Dinâmica Refatora Aí
🔧 Refatora Aí — do Estruturado para o Orientado a Objetos
⏱️ Duração: 15 minutos
👥 Agrupamento: grupos de 4 a 5 alunos
🎯 Objetivo: consolidar a transição do paradigma estruturado para o orientado a objetos por meio da refatoração de um trecho de código — desenvolvendo a capacidade de reconhecer oportunidades de melhoria estrutural e aplicar os conceitos de OO em uma situação concreta
1
Apresentação do código estruturado (2 min) — O professor projeta um trecho de código estruturado funcional mas desorganizado. O grupo precisa identificar: quais dados poderiam virar atributos de uma classe · quais funções poderiam virar métodos · qual seria o nome da classe · se haveria necessidade de herança
2
Refatoração em grupo (8 min) — Os grupos reescrevem a estrutura do código em OO na ficha: nome da classe, atributos com tipos e modificadores, assinatura dos métodos. Não é necessário escrever o código completo — apenas a estrutura
3
Apresentação e comparação (5 min) — Os grupos apresentam suas propostas de refatoração. O professor compara as estruturas e discute as diferenças de escolha
Elemento do código estruturadoVira em OOResultado na classe Produto
String nomeProduto · double precoProduto · int quantidadeEstoque Atributos private private String nome; private double preco; private int quantidadeEstoque;
void exibirProduto(nome, preco, qtd) Método sem parâmetros void exibirDados() — usa os próprios atributos
void aplicarDesconto(preco, percentual) Método com 1 parâmetro void aplicarDesconto(double percentual)
void atualizarEstoque(qtdAtual, qtdVendida) Método com 1 parâmetro void atualizarEstoque(int qtdVendida)
Criação e inicialização das variáveis soltas Construtor + getters/setters new Produto("Notebook", 3500.00, 10)
💡 Reforço pedagógico: refatoração é uma habilidade real e valorizada no mercado — desenvolvedores passam boa parte do tempo melhorando código existente, não apenas escrevendo código novo. Saber identificar quando e como refatorar é o que diferencia um desenvolvedor júnior de um pleno. Na próxima SA o sistema dá o próximo passo profissional: conexão com banco de dados.
📝 Avaliação Questionário · 10 questões
Parte 1 — Questões Objetivas (5 questões)
  • Nível alto · 4 alternativas: A) B) C) D)
  • Gabarito alternado, sem repetição consecutiva de letras
  • Temas: causa do NullPointerException, erro de acesso a atributo private, papel da classe abstrata no projeto, polimorfismo com array Pessoa[], vantagens da refatoração para OO
Parte 2 — Questões Discursivas (5 questões)
  • Cada questão possui título explicativo antes do enunciado
  • Sem linhas — apenas espaço em branco com R:
  • Temas: declarar a hierarquia Pessoa → Aluno com setter validado, identificar e corrigir um erro comum de OO em trecho de código, explicar o polimorfismo observado no array do projeto, refatorar um trecho estruturado para classe com atributos e métodos, justificar as vantagens da versão OO sobre a estruturada
Gabarito
  • Em página separada — uso exclusivo do professor
  • Objetivas: letra correta + justificativa completa
  • Discursivas: resposta esperada com critérios de correção e exemplos de código válidos
