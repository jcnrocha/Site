# UC: Programação de Aplicativos
## SA 33 — 3º Trimestre

---

## Tema

Projeto Final — Apresentação e Briefing

---

## Objetivos de Aprendizagem

Ao final da SA, o aluno será capaz de:

- Compreender os requisitos, critérios de avaliação e entregas esperadas do Projeto Final
- Formar grupos de trabalho e definir o tema do aplicativo a ser desenvolvido
- Elaborar o escopo inicial do projeto com base no Business Model Canvas
- Identificar as tecnologias, ferramentas e estrutura de classes necessárias para o projeto
- Planejar as etapas de desenvolvimento distribuídas nas SAs 34, 35 e 36

**Capacidades da matriz:** Integração de todas as capacidades da UC · Canvas · Aplicar os princípios de organização do trabalho no planejamento · Capacidades socioemocionais: organização, planejamento e trabalho em equipe

---

## Conteúdo

- Contextualização: *"As consolidações estão feitas — hoje começa o Projeto Final. Tudo o que aprendemos ao longo do ano será aplicado aqui: OO, banco de dados, técnicas de programação, testes, qualidade e Canvas"*

- **O que é o Projeto Final:**
  - Desenvolvimento de um aplicativo funcional com banco de dados em grupo
  - Integra todos os conteúdos da UC em um produto real e demonstrável
  - Avaliado em duas dimensões: **produto** (o sistema funcionando) e **processo** (como foi desenvolvido)
  - Entregável final: sistema rodando + Canvas + apresentação oral + repositório no GitHub

- **Requisitos obrigatórios do Projeto Final:**

  | Requisito | Descrição |
  |-----------|-----------|
  | Linguagem de programação | Linguagem da UC com paradigma orientado a objetos |
  | Mínimo de classes | Ao menos 3 classes com herança e encapsulamento |
  | Banco de dados | CRUD completo (INSERT, SELECT, UPDATE, DELETE) com Prepared Statement |
  | Testes unitários | Ao menos 5 testes cobrindo métodos principais |
  | Documentação | Javadoc/docstring nos métodos públicos |
  | Repositório | Projeto no GitHub com histórico de commits descritivos |
  | Canvas | Business Model Canvas preenchido para o aplicativo |
  | Apresentação | Pitch oral de 5 minutos com demonstração do sistema funcionando |

- **Critérios de avaliação:**

  | Critério | Peso | O que será avaliado |
  |----------|------|---------------------|
  | Sistema funcionando | 30% | CRUD completo · sem erros críticos em execução |
  | Qualidade do código | 25% | OO correta · DRY · nomes descritivos · documentação |
  | Testes unitários | 15% | Quantidade · cenários de borda · cobertura |
  | Canvas | 15% | 9 blocos preenchidos · coerência entre negócio e sistema |
  | Apresentação | 15% | Clareza · demonstração ao vivo · postura profissional |

- **Temas sugeridos para o Projeto Final:**

  | Tema | Descrição resumida |
  |------|--------------------|
  | Sistema de agendamento | Cadastro de clientes, serviços e horários com consulta e cancelamento |
  | Controle de estoque | Cadastro de produtos, entradas, saídas e relatório de saldo |
  | Sistema de biblioteca | Cadastro de livros, usuários, empréstimos e devoluções |
  | Gestão de tarefas | Cadastro de projetos, tarefas com status e responsável |
  | Sistema de vendas | Cadastro de produtos, clientes, pedidos e total da venda |
  | Clínica veterinária | Cadastro de animais, tutores, consultas e histórico |

  > Os grupos podem propor tema próprio — sujeito à aprovação do professor com base na viabilidade técnica

- **Canvas do Projeto Final — preenchimento inicial:**
  - Cada grupo preenche um Canvas básico do sistema escolhido ainda nesta SA
  - Foco nos blocos mais críticos para o desenvolvimento: Segmento de Clientes · Proposta de Valor · Recursos Principais · Atividades-Chave
  - Canvas será refinado e entregue na SA 37

- **Planejamento das etapas de desenvolvimento:**

  | SA | Data | Entrega esperada |
  |----|------|-----------------|
  | SA 34 | 21/10/2026 | Estrutura do projeto · modelagem de classes · banco de dados criado · primeiros módulos |
  | SA 35 | 28/10/2026 | CRUD completo · tratamento de erros · documentação · testes dos módulos principais |
  | SA 36 | 04/11/2026 | Integração dos módulos · refatoração · rastreabilidade · ajustes finais |
  | SA 37 | 11/11/2026 | Canvas finalizado · preparação da apresentação · pitch ensaiado |
  | SA 38 | 18/11/2026 | Apresentações para a turma · demonstração ao vivo |

- **Organização dos grupos:**
  - Grupos de 3 a 4 alunos — definidos nesta SA
  - Cada grupo define: tema · nome do sistema · responsável por cada módulo · repositório criado no GitHub
  - Todos os membros devem ter commits no repositório — trabalho individual rastreável

## Organização do Tempo

| Etapa | Atividade |
|-------|-----------|
| Abertura | Chamada · apresentação do Projeto Final: *"Chegamos ao momento mais importante da UC — o Projeto Final. Tudo que aprendemos converge aqui"* |
| Briefing | Professor apresenta os requisitos obrigatórios, critérios de avaliação, cronograma de entregas e temas sugeridos com detalhamento de cada um |
| Formação dos grupos | Alunos formam os grupos e escolhem o tema · professor orienta e aprova as escolhas |
| Canvas inicial | Cada grupo preenche o Canvas básico do sistema escolhido com foco nos blocos principais |
| Planejamento | Cada grupo cria o repositório no GitHub · define a estrutura de pastas · distribui responsabilidades · registra as decisões no README inicial |
| Fechamento | Cada grupo apresenta em 1 minuto: tema escolhido, proposta de valor e divisão de responsabilidades · professor faz perguntas técnicas para validar a viabilidade |

---

## Dinâmica — Integrada à SA

> Esta SA não possui dinâmica separada — a própria SA é uma grande dinâmica prática de planejamento e organização do Projeto Final. A interação entre os grupos, a escolha do tema, o preenchimento do Canvas e a criação do repositório constituem a atividade central desta SA.

---

## Avaliação

- Esta SA não possui questionário individual — a avaliação é o próprio entregável do grupo:
  - Canvas inicial preenchido (os 9 blocos com pelo menos os 4 blocos principais completos)
  - Repositório criado no GitHub com README descrevendo o sistema, os integrantes e o escopo
  - Divisão de responsabilidades documentada
- Avaliação formativa: professor registra observações sobre organização, clareza do escopo e viabilidade técnica de cada grupo
