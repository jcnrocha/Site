As Três Frentes da Aula

🎛️
Controle de Qualidade
"Agir sobre os desvios detectados para corrigi-los e prevenir recorrência — monitoramento sem controle é inútil"
  • Garante que o código atende aos padrões antes de chegar ao usuário
  • Responsabilidade de todo o time, não de um testador separado
  • Critérios de aceite no formato "Dado [contexto], quando [ação], então [resultado]"
  • Checklist de entrega verificado antes de cada Pull Request
📋
Registro de Não Conformidades
"Não conformidade é qualquer situação em que o produto ou processo não atende ao padrão — registrar é a base da melhoria"
  • O que registrar: descrição · quando e onde · impacto · causa raiz · ação corretiva · responsável e prazo
  • Permite rastrear histórico e identificar padrões recorrentes
  • Ferramentas: GitHub/GitLab Issues, Jira, Trello, Notion
  • Chamado claro economiza tempo da equipe inteira
🔄
Melhoria Contínua — PDCA
"Ciclo completo: monitorar → detectar desvio → controlar → registrar → melhorar"
  • P (Plan): definir padrões, critérios de aceite, checklist
  • D (Do): desenvolver seguindo os padrões
  • C (Check): monitorar, controlar, registrar não conformidades
  • A (Act): corrigir desvios e melhorar o processo para a próxima iteração

Cinco Práticas de Controle de Qualidade

👀
Code Review
Outro desenvolvedor analisa o código antes do merge — a cada Pull Request
✅
Critérios de Aceite
Lista do que a funcionalidade deve fazer para ser "pronta" — definidos antes do desenvolvimento
📝
Checklist de Entrega
Verificações obrigatórias antes de enviar o código — antes de abrir o Pull Request
🤖
Análise Estática
Ferramenta automática detecta problemas sem executar — a cada commit (pipeline CI)
🔁
Testes de Regressão
Reexecutar todos os testes ao integrar novo código — a cada merge na branch principal

Monitoramento × Controle — Qual a Diferença

📡
Monitoramento — detectar
  • Observar o que está acontecendo em tempo real
  • Detectar desvios: uptime caindo, erros subindo, lentidão
  • Dashboards, alertas e logs (conteúdo da SA 26)
  • Responde: "algo está errado?"
Detectar um problema e não agir não resolve nada
🎛️
Controle — agir
  • Agir sobre os desvios detectados para corrigi-los
  • Prevenir recorrência atacando a causa raiz
  • Code review, critérios de aceite, checklist, regressão
  • Responde: "o que fazemos para corrigir e não repetir?"
Ciclo completo: monitorar → detectar → controlar → registrar → melhorar

Ferramentas de Registro em TI

FerramentaUso
GitHub Issues / GitLab Issues Registro de bugs, melhorias e tarefas diretamente no repositório
Jira Gestão de projetos e rastreamento de bugs em empresas
Trello Kanban visual para equipes menores
Notion Documentação e registro de ocorrências de forma estruturada
✅ Checklist de qualidade antes do Pull Request (trabalhado em aula): o código compila sem erros · os testes existentes continuam passando · foram escritos testes para o novo código · a cobertura não diminuiu · o código está formatado no padrão · os métodos públicos estão documentados · não há código comentado desnecessariamente · não há credenciais ou senhas no código · o PR tem descrição clara · o cenário principal e os casos de borda foram testados manualmente. Exemplo de critério de aceite: "Dado que o usuário erra a senha 3 vezes consecutivas, quando tenta a quarta vez, então a conta é bloqueada por 15 minutos."

📮 Abre o Chamado — Registro Estruturado de Não Conformidades
⏱️ Duração: 15 minutos
👥 Agrupamento: grupos de 4 a 5 alunos
🎯 Objetivo: desenvolver a habilidade de registrar não conformidades de forma estruturada e objetiva — simulando o processo real de abertura de chamado/issue que desenvolvedores e analistas realizam diariamente em ferramentas como GitHub Issues e Jira
1
Apresentação do cenário (1 min por rodada) — O professor descreve um problema encontrado em um sistema fictício. O grupo deve registrar a não conformidade de forma estruturada na ficha, preenchendo todos os campos obrigatórios
2
Registro em grupo (9 min) — Os grupos preenchem a ficha de registro para cada cenário apresentado, incluindo: descrição, impacto, causa raiz provável, ação corretiva sugerida e prioridade
3
Comparação e debate (5 min) — Os grupos compartilham seus registros. O professor compara a qualidade dos registros e discute: "Um desenvolvedor que receber este chamado consegue entender e resolver o problema sem precisar perguntar nada?"
Cenário de não conformidadeO que registrar
Sistema de cadastro aceita e-mail sem o símbolo "@" Descrição · impacto (dados inválidos no banco) · causa provável (falta de validação) · ação (adicionar validação de e-mail antes do INSERT)
Página de relatório demora 12 segundos para carregar Descrição · impacto (experiência ruim do usuário) · causa provável (consulta SQL sem índice) · ação (analisar query e adicionar índice na coluna de busca)
Sistema exibe mensagem de erro com stack trace para o usuário final Descrição · impacto (exposição de informação técnica sensível) · causa provável (exceção não tratada) · ação (tratar exceção e exibir mensagem amigável)
Método calcularDesconto retorna valor negativo quando desconto > 100% Descrição · impacto (erro de cálculo em pedidos) · causa provável (falta de validação do percentual) · ação (validar percentual entre 0 e 100 antes do cálculo)
Campo da ficha de registroPreenchimento
TítuloResumo curto e objetivo do problema
DescriçãoO que acontece, em qual funcionalidade, como reproduzir
ImpactoQuem é afetado e qual a consequência
Causa raiz provávelO que no código ou processo causou o problema
Ação corretiva sugeridaO que deve ser feito para resolver
Prioridade · ResponsávelCrítica / Alta / Média / Baixa · quem deveria resolver
💡 Reforço pedagógico: um bom registro de não conformidade economiza tempo da equipe inteira — um chamado vago como "sistema não funciona" gera ciclos de perguntas e respostas que atrasam a resolução. Desenvolvedores que abrem chamados claros e bem documentados são muito mais valorizados porque respeitam o tempo dos colegas.

  • Temas: explicar o ciclo monitorar → controlar → registrar → melhorar, escrever dois critérios de aceite para a funcionalidade de login, registrar uma não conformidade completa a partir de um cenário, montar um checklist de entrega com ao menos 6 itens, aplicar o PDCA a uma situação de bug recorrente
Gabarito
  • Em página separada — uso exclusivo do professor
  • Objetivas: letra correta + justificativa completa
  • Discursivas: resposta esperada com critérios de correção
