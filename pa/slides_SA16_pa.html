Formatação — disciplina técnica, não estética

📐 Por que Formatação Importa

"Código é lido muito mais vezes do que é escrito" — formatação não é preferência pessoal, é disciplina técnica que afeta diretamente a velocidade de manutenção e a qualidade do trabalho em equipe

IMPACTO NA EQUIPE Guias de Estilo Obrigatórios
Empresas adotam padrões obrigatórios: Google Java Style · PEP 8 (Python) · Airbnb JS Guide · toda a equipe segue o mesmo padrão para que qualquer membro consiga ler qualquer parte do código
IMPACTO NO CÓDIGO Legibilidade = Manutenção
Código bem formatado revela erros mais facilmente · a estrutura visual ajuda a entender o fluxo · código que parece confuso É confuso — reformatar faz parte do trabalho profissional

Código é lido por pessoas — compiladores não ligam para formatação, mas seus colegas (e você no futuro) precisam entender o código sem esforço

Antes × Depois — mesma lógica, resultado diferente

💻 Código Desformatado × Código Bem Formatado

Os dois trechos têm a mesma lógica — a diferença está em nomes descritivos, indentação e espaçamento que tornam o código legível ou ilegível

✗ Desformatado — difícil de ler
public double calc(double a,double b,double c){ double m=(a+b+c)/3; if(m>=7){return m;} else{return -1;}}
✔ Bem formatado — fácil de ler
public double calcularMedia( double nota1, double nota2, double nota3) { double media = (nota1 + nota2 + nota3) / 3; // Critério: >= 7 aprovado if (media >= 7) { return media; } return -1; }

A diferença: nomes descritivos (calc → calcularMedia · a,b,c → nota1,nota2,nota3) + indentação + espaçamento = código que se explica

VS Code — Shift+Alt+F

🔧 Prettier — Formatação Automática

O Prettier analisa o código e aplica os padrões de formatação automaticamente — atalho Shift+Alt+F no VS Code · configurar editor.formatOnSave para formatar ao salvar

ATIVAR NO VS CODE editor.formatOnSave
Ctrl+, → Settings · buscar "Format On Save" · ativar · o Prettier formata automaticamente cada vez que Ctrl+S é pressionado · manter o padrão sem esforço manual
LIMITAÇÃO IMPORTANTE A ferramenta não substitui o conhecimento
Prettier cuida da indentação, espaçamento e chaves · não muda nomes de variáveis · não adiciona comentários · não renomeia calc() para calcularMedia() — isso é responsabilidade do programador

Atalho manual: Shift+Alt+F formata o arquivo atual · Shift+Alt+F em seleção formata apenas o trecho selecionado · Prettier formata, mas você decide os nomes

Comentando com propósito

💬 Comentários — O Porquê, Não o Quê

Comentário bom explica o porquê de uma decisão de implementação — não repete o que o código já diz · código bem nomeado dispensa comentários óbvios

❌ COMENTÁRIO INÚTIL — RUÍDO Repete o que o código já diz
// Incrementa i em 1 → i++ · // Retorna true → return true · // Declara a variável nota · Esses comentários são ruído — o código já diz a mesma coisa com mais precisão · removê-los deixa o código mais limpo
✔ COMENTÁRIO ÚTIL — PORQUÊ Explica decisão não óbvia
// Percorre de trás pra frente para evitar reindexação · // Divide por 1024 para converter bytes em KB · // Critério: >= 7 aprovado, >= 5 recuperação · esses explicam a lógica por trás da implementação

Comentário que explica o que o código já diz é ruído — se o nome da variável ou do método foi bem escolhido, o comentário muitas vezes não é necessário

Exemplos práticos de comentário

💻 Comentário Inútil × Comentário Útil

Comentários inúteis poluem o código sem acrescentar informação — e quando o código muda, o comentário errado que não foi atualizado passa a enganar quem lê

✗ Comentários inúteis — ruído
// Declara a variável nota double nota = 0; // Verifica se maior ou igual a 7 if (nota >= 7) { // Retorna verdadeiro return true; }
✔ Comentário útil — porquê
double nota = 0; // Critério: >= 7.0 definido pelo // Regimento Escolar §12 if (nota >= 7) { return true; }

A versão correta tem menos e melhores comentários — apenas o que explica a decisão de negócio por trás do critério, que não seria óbvio para quem lê o código sem contexto

A melhor documentação é o nome certo

🏷️ Nomenclatura — Código que se Explica

Nomear bem variáveis e métodos é a forma mais valiosa de documentação — código que se explica pelo nome precisa de muito menos comentário e é lido com muito mais facilidade

JAVA — GOOGLE JAVA STYLE camelCase + PascalCase
Variáveis e métodos: calcularMedia() · notaFinal · listaDeTurmas · Classes: class Aluno · class CalculadoraDeNotas · Constantes: NOTA_MINIMA_APROVACAO · MAX_ALUNOS
PYTHON — PEP 8 snake_case + PascalCase
Variáveis e funções: calcular_media() · nota_final · lista_turmas · Classes: class Aluno · class CalculadoraNotas · Constantes: NOTA_MINIMA · MAX_ALUNOS

Renomear calc(a, b, c) para calcularMedia(nota1, nota2, nota3) é a maior melhoria possível — o método passa a se explicar completamente sem precisar de comentário

Documentação formal de classes e métodos

📖 Documentação Formal — Javadoc e Docstrings

Javadoc e docstrings documentam o contrato público de classes e métodos — descrevem o que fazem, parâmetros e retorno — lidos por quem usa a classe sem ver o código-fonte

JAVA — /** JAVADOC */ Tags: @param · @return · @throws
Fica antes da declaração do método · @param nota1 primeira nota (0.0 a 10.0) · @param nota2 segunda nota · @return média aritmética ou -1 se inválida · @throws IllegalArgumentException · @author e @version na classe
PYTHON — """DOCSTRING""" Padrão: Args / Returns / Raises
"""Calcula a média das notas.""" imediatamente após def · Args: nota1 (float): primeira nota (0.0–10.0) · nota2 (float): segunda nota · Returns: float: média ou -1 · Raises: ValueError: se nota inválida

Fica imediatamente antes da declaração do método ou classe · permite gerar documentação HTML automaticamente — Javadoc tool em Java · Sphinx em Python

Quando usar cada tipo

⚖️ Comentário // × Documentação /** */

💬 Comentário — // e /* */
Onde: dentro do método, junto à lógica
Explica o PORQUÊ de uma decisão interna
Visível apenas no código-fonte
Lido por: quem está no código-fonte
Comentário que repete o código = ruído
📖 Javadoc — /** */ e docstrings
Onde: antes de classes e métodos públicos
Descreve O QUÊ o método faz, params e retorno
Gera documentação HTML automática
Lido por: quem usa a classe sem ver o código
Tags: @param · @return · @throws · @author

Use // para registrar decisões internas não óbvias · use /** */ para documentar o contrato público de métodos e classes que outros desenvolvedores vão usar
