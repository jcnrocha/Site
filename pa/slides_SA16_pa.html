Slide 05 · Problematização
O mesmo código — duas versões. Qual você prefere manter?
❌ Versão A — como entregue
public double calc(double a,double b,double c){ double r=0; if(a>=0&&b>=0&&c>=0){r=(a+b+c)/3;} if(r>=7.0){System.out.println("aprovado");} else if(r>=5.0){System.out.println("recup");} else{System.out.println("reprovado");} return r;}
✅ Versão B — profissional
/** * Calcula a média e exibe o resultado do aluno. * @param nota1 Primeira nota (0.0 a 10.0) * @param nota2 Segunda nota (0.0 a 10.0) * @param nota3 Terceira nota (0.0 a 10.0) * @return Média aritmética das três notas */ public double calcularMedia(double nota1, double nota2, double nota3) { double media = (nota1 + nota2 + nota3) / 3.0; // Critério: >= 7 aprovado, >= 5 em recuperação if (media >= 7.0) { System.out.println("Aprovado"); } else if (media >= 5.0) { System.out.println("Em recuperação"); } else { System.out.println("Reprovado"); } return media; }
As duas versões fazem a mesma coisa. A diferença é que daqui a 6 meses a Versão B pode ser lida, mantida e evoluída por qualquer desenvolvedor da equipe — inclusive por você mesmo — sem precisar adivinhar o que o código faz.
Slide 06 · Formatação
Cinco Padrões Essenciais de Formatação
📏
Indentação
2 ou 4 espaços por nível · nunca misturar espaços e tabs · blocos internos sempre recuados
🧱
Chaves e Blocos
Abertura na mesma linha do comando · fechamento em linha própria · linha em branco entre métodos
↔️
Espaçamento
Espaço antes e depois de operadores · espaço após vírgula: a + b · f(a, b)
📃
Limite de Linha
Máximo 80 a 120 caracteres por linha · linha longa: quebrar em múltiplas linhas com indentação
🏷️
Nomenclatura
camelCase variáveis e métodos · PascalCase classes · snake_case Python: calcular_media
Guias de estilo obrigatórios no mercado: Google Java Style Guide (Java) · PEP 8 (Python) · Airbnb JavaScript Style Guide (JS/TS) · Essas regras não são sugestões — equipes profissionais as aplicam em todo commit via CI/CD.
Código é lido muito mais vezes do que é escrito. Um desenvolvedor lê código 10× mais do que escreve. Formatação padronizada reduz o custo cognitivo de cada leitura — e o custo se acumula diariamente em toda a equipe.
Slide 07 · Indentação
Indentação e Blocos — Antes e Depois
❌ Sem indentação
public void verificarAluno(String nome, double nota){ if(nota>=7.0){System.out.println(nome+" aprovado");} else if(nota>=5.0){System.out.println(nome+" recuperação");} else{System.out.println(nome+" reprovado");} System.out.println("Processado.");}
✅ Formatação correta
public void verificarAluno(String nome, double nota) { if (nota >= 7.0) { System.out.println(nome + " aprovado"); } else if (nota >= 5.0) { System.out.println(nome + " recuperação"); } else { System.out.println(nome + " reprovado"); } System.out.println("Processado."); }
Chave de abertura na mesma linha do comando: if (cond) { · nunca em linha separada
Chave de fechamento sempre em linha própria, no mesmo nível do comando que a abriu: }
Linha em branco entre métodos distintos · nunca entre o comando e sua chave de abertura

Slide 10 · Comentários
Comentários — Explica o PORQUÊ, Não o QUÊ
Regra fundamental: bons nomes de variáveis e métodos já explicam o que o código faz. O comentário existe para registrar por que uma decisão foi tomada — uma restrição oculta, uma lógica não óbvia, um comportamento que surpreenderia quem lê.
✅ Comentários que agregam valor
// Percorre de trás para frente para evitar // reindexação ao remover elementos da lista for (int i = lista.size() - 1; i >= 0; i--) { ... } // Divide por 8 para converter Mbps em MB/s double velocidade = mbps / 8.0; // Critério da escola: >= 7 aprovado, >= 5 recuperação if (media >= 7.0) { ... }
❌ Comentários que são ruído
// Incrementa i em 1 i++; // Seta o nome do aluno aluno.setNome(nome); // Cria um PreparedStatement PreparedStatement ps = conn.prepareStatement(sql); // Retorna a nota return nota;
Teste prático: se o comentário apenas repete o que o código já diz com palavras diferentes, apague-o. Código com menos ruído é mais fácil de ler do que código com muitos comentários inúteis.
Slide 11 · Comentários
Tipos de Comentários — Linha e Bloco
Java — linha // e bloco /* */
// Comentário de uma linha — decisão específica double velocidade = mbps / 8.0; // Mbps → MB/s /* * Comentário de bloco — lógica mais complexa: * ordena pelo campo 'prioridade' antes de processar * porque a API downstream exige items em ordem */ lista.sort(Comparator.comparing(Item::getPrioridade));
Python — linha # e bloco # múltiplas linhas
# Comentário de uma linha velocidade = mbps / 8.0 # Mbps → MB/s # Python não tem bloco /* */ nativo # Use múltiplas linhas com # para comentários longos: # Ordena pelo campo 'prioridade' antes de processar # porque a API downstream exige items em ordem lista.sort(key=lambda x: x.prioridade)
✓ Quando comentar
  • Constante ou fórmula com regra de negócio implícita: / 8.0 converte unidade
  • Escolha não óbvia de algoritmo: percorrer de trás pra frente
  • Dependência com sistema externo ou comportamento estranho da lib
  • Critério de aprovação definido externamente (pela escola, regras fiscais etc.)
✗ Não comentar
  • O que bons nomes já explicam: calcularMedia() dispensa comentário
  • Operações triviais: i++ · setNome(nome) · return nota
  • Código morto — simplesmente apague; git guarda o histórico
  • Comentários que serão desatualizados e ficarão mentindo
Slide 12 · Documentação Formal
Javadoc — Documentação Formal em Java ( /** */ )
Javadoc completo
/** * Busca um aluno pelo identificador único no banco. * * @param id Identificador do aluno (gerado pelo BD) * @return Objeto Aluno encontrado, ou null se não existir * @throws SQLException Se ocorrer erro de conexão * @author Prof. Julio Cesar * @version 1.0 */ public Aluno buscarPorId(int id) throws SQLException { String sql = "SELECT * FROM alunos WHERE id = ?"; // implementação... }
Tags Javadoc
@param nome — descreve um parâmetro do método: nome e o que representa
@return — descreve o valor retornado pelo método (omitir para void)
@throws — exceção que o método pode lançar e em que condição
@author e @version — identificação do autor e versão do método/classe
Onde fica: imediatamente antes da declaração da classe ou do método · Quem lê: quem vai usar a classe sem ver o código-fonte — um colega de equipe, você mesmo daqui a 6 meses · Diferença dos comentários: o Javadoc documenta o contrato público, não a implementação interna.
Slide 13 · Documentação Formal
Docstrings — Documentação Formal em Python ( """ """ )
Docstring completa (estilo Google)
def calcular_media(nota1: float, nota2: float) -> float: """ Calcula a média aritmética de duas notas. Args: nota1: Primeira nota do aluno (0.0 a 10.0) nota2: Segunda nota do aluno (0.0 a 10.0) Returns: Média aritmética entre nota1 e nota2 Raises: ValueError: Se alguma nota for negativa """ if nota1 < 0 or nota2 < 0: raise ValueError("Notas não podem ser negativas") return (nota1 + nota2) / 2.0
Acessando a documentação
# No Python REPL ou notebook: help(calcular_media) # Exibe a docstring formatada # Também acessível por: print(calcular_media.__doc__) # No VS Code: hover sobre a função # exibe a docstring como tooltip
Args: lista de parâmetros com nome e descrição
Returns: o que o método retorna e o tipo
Raises: exceções possíveis e condições que as ativam
Boa prática Python: toda função pública deve ter docstring — é a forma como o help() do interpretador e as ferramentas de autocompletar (VS Code, PyCharm) exibem a documentação ao usuário da função.
Slide 14 · Comparativo
Comentário de Explicação × Documentação Formal
💬 Comentário — // e /* */
  • Onde fica: dentro do método, junto da lógica
  • Explica: o PORQUÊ de uma decisão de implementação
  • Quem lê: quem está lendo o código-fonte internamente
  • Tom: técnico e contextual — para desenvolvedores
  • Regra: se o código já é claro, não comente. Menos é mais.
📖 Javadoc e Docstrings
  • Onde fica: antes da declaração da classe ou método
  • Descreve: o QUE o método faz, parâmetros e retorno
  • Quem lê: quem usa a classe sem precisar ver o código
  • Tom: contrato público — para usuários do método/classe
  • Regra: todo método e classe pública merece documentação.
SituaçãoUse
Explicar por que se divide por 8 na conversão de velocidade Comentário //
Descrever parâmetros e retorno de um método público de cálculo Javadoc / Docstring
Registrar critério externo de aprovação (regra da escola) Comentário //
Documentar uma classe de modelo (Aluno, Produto, Pedido) Javadoc / Docstring
Slide 15 · Exemplo Completo
calcularMedia — Documentado em Java e Python
Java — Javadoc + comentário
/** * Calcula a média aritmética de duas notas. * * @param nota1 Primeira nota do aluno (0.0 a 10.0) * @param nota2 Segunda nota do aluno (0.0 a 10.0) * @return Média entre nota1 e nota2 */ public double calcularMedia(double nota1, double nota2) { // Divisão por 2.0 garante resultado decimal, não inteiro return (nota1 + nota2) / 2.0; }
Python — docstring + comentário
def calcular_media(nota1: float, nota2: float) -> float: """ Calcula a média aritmética de duas notas. Args: nota1: Primeira nota do aluno (0.0 a 10.0) nota2: Segunda nota do aluno (0.0 a 10.0) Returns: Média aritmética entre nota1 e nota2 """ # float() garante divisão decimal em Python 2 legado return (nota1 + nota2) / 2.0

Slide 16 · Dinâmica
Formata e Documenta — Transformando Código Bagunçado em Código Profissional
📐
Dinâmica em Grupo · 15 minutos · Grupos de 4–5 alunos
Receber o código abaixo e aplicar: formatação padrão · nomes descritivos · comentário útil · Javadoc ou docstring
Código original — reescreva este
public double calc(double a,double b,double c){ double r=0; if(a>=0&&b>=0&&c>=0){r=(a+b+c)/3;} if(r>=7.0){System.out.println("aprovado");} else if(r>=5.0){System.out.println("recuperacao");} else{System.out.println("reprovado");} return r;}
O que cada grupo deve fazer
  • Reformatar — indentação 4 espaços, chaves no padrão, espaço em operadores
  • Renomear — calc → calcularMedia · a, b, c → nota1, nota2, nota3 · r → media
  • Comentar — adicionar // Critério: >= 7 aprovado, >= 5 recuperação no lugar certo
  • Documentar — adicionar Javadoc com @param para cada nota e @return da média
  • Avaliar — algum comentário ficaria desnecessário (ruído) no código reformatado?
Slide 17 · Dinâmica
Formata e Documenta — Problemas e Melhorias
Problema no código originalMelhoria aplicadaResultado esperado
Nome do método: calc Nome descritivo calcularMedia()
Parâmetros: a, b, c Nomes significativos nota1, nota2, nota3
Variável de retorno: r Nome significativo media
Tudo na mesma linha, sem indentação, chaves inline Formatação padrão Indentação 4 espaços · chave abertura na mesma linha · fechamento em linha própria
Critério de aprovação implícito nos ifs Comentário do porquê // Critério: >= 7 aprovado, >= 5 recuperação
Nenhuma documentação do método Javadoc /** @param nota1 ... @return ... */
Reforço pedagógico: renomear calc, a, b, c e r para nomes descritivos é a forma mais valiosa de documentação. Código que se explica pelo nome precisa de muito menos comentário — e nunca vai desatualizar.
Slide 18 · Resumo
O que aprendemos hoje
📐
Formatação Padronizada
Indentação · chaves no padrão · espaçamento em operadores · limite de 80–120 chars · nomenclatura camelCase/PascalCase. Guias de estilo são obrigatórios em equipes profissionais.
✨
Prettier e Ferramentas
Shift+Alt+F ou formatOnSave automatizam a formatação. Aprenda as regras: a ferramenta ajuda, mas não substitui o conhecimento.
💬
Comentários ( // e /* */ )
Explica o PORQUÊ, não o quê. Registra decisões não óbvias, restrições ocultas e critérios externos. Comentário que repete o código é ruído — apague.
📖
Javadoc e Docstrings
Documentação formal para quem usa a classe sem ver o código. Tags: @param · @return · @throws. Todo método público merece documentação.
