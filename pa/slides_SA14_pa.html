Slide 05 · CREATE
INSERT — Inserindo Dados no Banco
Java (JDBC)
String sql = "INSERT INTO alunos (nome, nota) VALUES (?, ?)"; try (Connection conn = Conexao.getConexao(); PreparedStatement ps = conn.prepareStatement(sql)) { ps.setString(1, nome); ps.setDouble(2, nota); ps.executeUpdate(); conn.commit(); System.out.println("Aluno cadastrado!"); }
Python (sqlite3)
sql = "INSERT INTO alunos (nome, nota) VALUES (?, ?)" cursor.execute(sql, (nome, nota)) conexao.commit() # confirma a inserção print("Aluno cadastrado!")
📝
Sintaxe SQL
INSERT INTO tabela (col1, col2)
VALUES (v1, v2)
💾
commit() obrigatório
Sem commit(), os dados são descartados ao fechar a conexão — nunca esqueça
⚙️
executeUpdate
Usado para INSERT, UPDATE e DELETE — retorna o número de linhas afetadas
Slide 06 · READ
SELECT — Consultando Dados do Banco
SELECT — todos os registros
String sql = "SELECT * FROM alunos"; PreparedStatement ps = conn.prepareStatement(sql); ResultSet rs = ps.executeQuery(); while (rs.next()) { String nome = rs.getString("nome"); double nota = rs.getDouble("nota"); System.out.println(nome + " — " + nota); }
SELECT — com filtro WHERE
String sql = "SELECT nome, nota FROM alunos" + " WHERE nota >= ?"; PreparedStatement ps = conn.prepareStatement(sql); ps.setDouble(1, 7.0); ResultSet rs = ps.executeQuery(); while (rs.next()) { System.out.println(rs.getString("nome")); }
SELECT não exige commit() — leitura não modifica dados. Use executeQuery() para SELECT. Percorra o resultado com while(rs.next()) em Java ou for row in cursor em Python. Prefira selecionar apenas as colunas necessárias — evite SELECT * em produção.
Slide 07 · Filtros
WHERE — Filtrando Dados na Consulta
OperadorExemplo no SQLO que faz
= WHERE nome = ? Registros com nome exatamente igual ao valor
>= / <= WHERE nota >= ? Registros com nota maior ou igual ao valor
!= WHERE status != ? Registros com status diferente do valor
LIKE WHERE nome LIKE ? → valor: "%Ana%" Nomes que contêm "Ana" em qualquer posição
AND / OR WHERE nota >= ? AND turma = ? Combina múltiplas condições
⚠️ LIKE com Prepared Statement: o % vai dentro do valor, não no SQL. Exemplo Java: ps.setString(1, "%" + busca + "%") · nunca escreva LIKE '%?%' — o ? precisa estar solto no SQL.
Slide 08 · Segurança
Concatenação × Prepared Statement
❌ ERRADO — Concatenação direta
String sql = "INSERT INTO alunos VALUES ('" + nome + "', " + nota + ")"; // Vulnerável a SQL Injection // Erros de tipo e de aspas // NUNCA faça isso — em nenhuma situação
✅ CORRETO — Prepared Statement
String sql = "INSERT INTO alunos VALUES (?, ?)"; PreparedStatement ps = conn.prepareStatement(sql); ps.setString(1, nome); ps.setDouble(2, nota); // Valores enviados separados do SQL // SGBD os trata como dados — nunca como código
⚠️
SQL Injection
Concatenação permite que o usuário insira código SQL malicioso como valor de entrada
🐛
Erros de Tipo
Concatenar double ou date no SQL gera erros de formato imprevisíveis
🛡️
Prepared Statement
SQL e valores chegam ao SGBD separados — impossível injetar código no SQL
Slide 09 · Vulnerabilidade
SQL Injection — O Ataque Mais Comum do Mundo
O que é: o usuário digita código SQL como se fosse um dado comum — e o sistema insere esse código diretamente na consulta, executando comandos não autorizados.
Ataque de login
Usuário digita no campo login: admin' OR '1'='1 SQL gerado (concatenação): SELECT * FROM usuarios WHERE login = 'admin' OR '1'='1' Resultado: acessa sem saber a senha!
Ataque destrutivo
Usuário digita no campo nome: '; DROP TABLE alunos; -- SQL gerado: INSERT INTO alunos VALUES (''); DROP TABLE alunos; --', ... Resultado: tabela apagada!
Prepared Statement é a defesa: o valor é enviado separado do SQL — o SGBD trata como dado puro, nunca como código executável. Usar Prepared Statement não é boa prática: é uma obrigação profissional em qualquer sistema que lida com dados de usuários.
Slide 10 · Tratamento de Erros
try / catch / finally — Lidando com Falhas
Java
try { PreparedStatement ps = conn.prepareStatement(sql); ps.setString(1, nome); ps.executeUpdate(); conn.commit(); } catch (SQLException e) { System.err.println("Erro: " + e.getMessage()); conn.rollback(); // desfaz se falhou } finally { conn.close(); // sempre fecha }
Python
try: cursor.execute(sql, (nome, nota)) conexao.commit() except Exception as e: print(f"Erro: {e}") conexao.rollback() # desfaz se falhou finally: cursor.close() conexao.close() # sempre fecha
🔄
try
Contém o código que pode falhar — a operação com o banco
🚨
catch / except
Captura o erro sem travar o programa — exibe a mensagem e faz rollback
🔒
finally
Sempre executado — garante que a conexão seja fechada mesmo com erro
Slide 11 · commit()
commit() — Por Que Ele é Obrigatório
❌ Sem commit()
cursor.execute(sql, (nome, nota)) # commit() esquecido! conexao.close() # Dados descartados — não persistiram # A transação foi revertida ao fechar
Os dados não são salvos — desaparecem quando a conexão fecha
✅ Com commit()
cursor.execute(sql, (nome, nota)) conexao.commit() # confirma! conexao.close() # Dados persistidos permanentemente
Os dados são salvos definitivamente — persistem mesmo após fechar a conexão
Regra: após qualquer INSERT, UPDATE ou DELETE → chame commit(). SELECT não precisa de commit — não modifica dados. Em Java com auto-commit=false, o commit é sempre manual e obrigatório.
Slide 12 · Fluxo
Fluxo Completo de uma Operação com Banco
🔌
1. Conectar
Abrir conexão com o banco de dados via driver
📝
2. Preparar
prepareStatement() com ? nos valores dinâmicos
⚙️
3. Parâmetros
setString(), setDouble() — valores reais nos ?
▶️
4. Executar
executeUpdate() ou executeQuery()
💾
5. commit()
Confirmar alteração — apenas para escrita
🔒
6. Fechar
close() no finally — liberar recursos
Sempre no finally: feche cursor e conexão. Em Java, prefira try-with-resources — fecha automaticamente sem finally explícito. Em Python, use with conexao: para o mesmo efeito.
Slide 13 · Exemplo
Exemplo Completo — Sistema de Alunos em Java
INSERT — Cadastrar aluno
String sql = "INSERT INTO alunos (nome, nota)" + " VALUES (?, ?)"; try (Connection conn = Conexao.getConexao(); PreparedStatement ps = conn.prepareStatement(sql)) { ps.setString(1, "Maria"); ps.setDouble(2, 8.5); ps.executeUpdate(); conn.commit(); System.out.println("Cadastrado!"); } catch (SQLException e) { System.err.println(e.getMessage()); }
SELECT — Buscar aprovados (nota ≥ 7)
String sql = "SELECT nome, nota FROM alunos" + " WHERE nota >= ?"; try (Connection conn = Conexao.getConexao(); PreparedStatement ps = conn.prepareStatement(sql)) { ps.setDouble(1, 7.0); ResultSet rs = ps.executeQuery(); while (rs.next()) { System.out.printf("%s — %.1f%n", rs.getString("nome"), rs.getDouble("nota")); } } catch (SQLException e) { System.err.println(e.getMessage()); }
Slide 14 · Boas Práticas
Boas Práticas de CRUD
✓ FAZ
  • Usar Prepared Statement sempre que houver valores dinâmicos no SQL
  • Chamar commit() após INSERT, UPDATE e DELETE
  • Fechar conexão e cursor no bloco finally
  • Validar os dados na aplicação antes de enviar ao banco
  • Usar try-with-resources em Java para fechamento automático
  • Selecionar apenas as colunas necessárias no SELECT
✗ EVITA
  • Concatenar valores diretamente no SQL — porta aberta para SQL Injection
  • Esquecer o commit() após operações de escrita
  • Deixar a conexão aberta indefinidamente — vazamento de recursos
  • Usar SELECT * em consultas de produção
  • Confiar apenas nas constraints do banco para validar dados
  • Usar catch vazio — erros silenciosos são os mais perigosos

Principais Métodos JDBC — Referência Rápida
MétodoUsoObservação
prepareStatement(sql) INSERT, SELECT, UPDATE, DELETE Sempre usar — nunca concatenar
setString(i, valor) Parâmetro String/texto i começa em 1, não em 0
setInt(i, valor) Parâmetro inteiro Para IDs e contadores
setDouble(i, valor) Parâmetro decimal Para notas, preços, valores
executeUpdate() INSERT, UPDATE, DELETE Retorna nº de linhas afetadas
executeQuery() SELECT Retorna ResultSet — percorrer com rs.next()

conn.commit() Confirmar transação Obrigatório após escrita no banco
conn.rollback() Desfazer transação Chamar no bloco catch se houver erro
