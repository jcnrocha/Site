CRUD — as quatro operações fundamentais

🗃️ O que é CRUD

CRUD é o acrônimo das quatro operações fundamentais de persistência em qualquer sistema de software — Create · Read · Update · Delete — toda funcionalidade de dados passa por uma ou mais delas

FOCO DESTA SA ➕ Create + 🔎 Read
INSERT adiciona novos registros ao banco · SELECT recupera e filtra dados com WHERE · são as operações mais usadas no dia a dia de um sistema
SA 15 — PRÓXIMA AULA ✏️ Update + 🗑️ Delete
UPDATE modifica registros existentes no banco · DELETE remove registros permanentemente · ambos serão cobertos na SA 15 em 27/05

commit() — obrigatório: toda operação de escrita (C, U e D) exige commit() para confirmar no banco — sem ele, a alteração é descartada mesmo sem erro

Visão geral · referência rápida

📊 As Quatro Operações do CRUD

Operação Comando SQL Status
C — Create (Inserir)INSERT INTO tabela (col) VALUES (?)Foco SA 14
R — Read (Consultar)SELECT col FROM tabela WHERE col = ?Foco SA 14
U — Update (Atualizar)UPDATE tabela SET col = ? WHERE id = ?SA 15
D — Delete (Remover)DELETE FROM tabela WHERE id = ?SA 15
commit() — Confirmar— confirma C, U e D no bancoObrigatório

Atenção: SELECT (R) não exige commit — apenas operações de escrita modificam dados e precisam ser confirmadas com commit()

Inserindo dados no banco

➕ INSERT — Como Inserir Registros

INSERT INTO cria um novo registro em uma tabela — sempre use Prepared Statement com marcadores ? e chame commit() para confirmar

01
Abrir conexão — DriverManager.getConnection(url, user, pass) em Java · sqlite3.connect(arquivo) em Python
02
Preparar o SQL com ? — INSERT INTO alunos (nome, nota) VALUES (?, ?) — os ? são marcadores para os valores
03
Vincular os valores — ps.setString(1, nome) · ps.setDouble(2, nota) — tipos explícitos por posição
04
Executar o INSERT — ps.executeUpdate() em Java · cursor.execute(sql, (nome, nota)) em Python — retorna linhas afetadas
05
Confirmar com commit() — con.commit() em Java · conexao.commit() em Python — sem commit a inserção é descartada
INSERT na prática — Java

💻 INSERT — Concatenação × Prepared Statement

Concatenar valores no SQL abre a porta para SQL Injection — uma das vulnerabilidades mais exploradas do mundo — e causa erros de tipo e de aspas

✗ Concatenação — ERRADO
// ⚠️ SQL Injection possível! String sql = "INSERT INTO alunos " + "(nome, nota) VALUES ('" + nome + "', " + nota + ")"; stmt.executeUpdate(sql); con.commit();
✔ Prepared Statement — CORRETO
String sql = "INSERT INTO alunos " + "(nome, nota) VALUES (?, ?)"; PreparedStatement ps = con.prepareStatement(sql); ps.setString(1, nome); ps.setDouble(2, nota); ps.executeUpdate(); con.commit(); // confirma

Prepared Statement: o SQL é pré-compilado e os valores chegam depois — o banco trata os dois de forma independente e nunca interpreta dados como código

Consultando dados no banco

🔎 SELECT — Como Recuperar Registros

SELECT recupera registros do banco — sem WHERE retorna todos · com WHERE filtra apenas os que atendem à condição — muito mais eficiente que buscar tudo e filtrar no código

SEM FILTRO SELECT * FROM alunos
Retorna todos os registros da tabela · útil para listar dados completos · atenção com tabelas grandes — pode ser lento e consumir muita memória
COM FILTRO WHERE SELECT ... WHERE nota >= ?
Retorna apenas registros que atendem à condição · filtro processado pelo banco de dados, não pelo código · muito mais eficiente e preciso

SELECT não exige commit() — a operação apenas lê dados do banco e nunca os modifica · apenas INSERT, UPDATE e DELETE precisam de commit

SELECT com WHERE — passo a passo

📋 SELECT com WHERE — Passo a Passo

Para filtrar registros com segurança, prepare o SQL com ? e defina os parâmetros no PreparedStatement — o banco aplica o filtro antes de retornar os dados

01
Preparar o SQL com filtro — SELECT nome, nota FROM alunos WHERE nota >= ? — o ? é o marcador do parâmetro
02
Criar PreparedStatement e vincular — PreparedStatement ps = con.prepareStatement(sql) · ps.setDouble(1, 7.0)
03
Executar a consulta — ResultSet rs = ps.executeQuery() — SELECT não usa commit · Python: cursor.execute(sql, (7.0,))
04
Percorrer o resultado — while (rs.next()) em Java · for row in cursor em Python — cada iteração é um registro retornado
05
Ler os valores de cada linha — rs.getString("nome") · rs.getDouble("nota") — Python: row[0], row[1] ou row["nome"]
Parâmetros seguros com marcadores ?

🛡️ Prepared Statement — Como Funciona

O Prepared Statement separa o SQL dos dados — o banco recebe os dois de forma independente e nunca confunde dado com comando SQL — essa separação é a proteção contra SQL Injection

JAVA · JDBC Marcador: ?
con.prepareStatement(sql) · ps.setString(1, nome) · ps.setDouble(2, nota) · ps.setInt(3, id) — tipo explícito por posição (começa em 1)
PYTHON · sqlite3 / mysql-connector Marcador: ? ou %s
cursor.execute(sql, (nome, nota)) — valores passados como tupla · o driver cuida dos tipos automaticamente · sqlite3 usa ? · mysql-connector usa %s

SQL Injection: quando um atacante digita código SQL em um campo de entrada para manipular o banco — ex: ' OR '1'='1 — Prepared Statement bloqueia esse ataque

Comparativo de abordagens

⚖️ Concatenação × Prepared Statement

✗ Concatenação — Evitar
✗ SQL e dados misturados na mesma string
✗ Vulnerável a SQL Injection
✗ Erros de tipo: String onde era int ou double
✗ Aspas faltando → erro de sintaxe SQL
✗ Comportamento imprevisível com dados do usuário
✔ Prepared Statement — Fazer
✔ SQL pré-compilado, valores enviados depois
✔ Imune a SQL Injection — proteção automática
✔ setString, setDouble, setInt — tipos explícitos
✔ Sem risco de erro de aspas ou sintaxe SQL
✔ Comportamento previsível e auditável

Prepared Statement não é apenas boa prática — é obrigatório em qualquer sistema que recebe dados de usuários · nunca concatenar valores diretamente no SQL

Lidando com falhas de banco

⚠️ Tratamento de Erros com try/catch

Operações com banco podem falhar: BD indisponível, constraint violada, SQL inválido — o try/catch evita que o programa trave e garante que os recursos sejam fechados

01
try { } — envolve toda a operação: prepareStatement, setXxx, executeUpdate/Query e commit
02
catch (SQLException e) — captura falhas específicas do banco: BD fora do ar, constraint violada, SQL inválido
03
catch (Exception e) — captura outros erros inesperados que não sejam diretamente do SQL
04
finally { } — sempre executa, com erro ou sem: fecha cursor, PreparedStatement e conexão · sem finally = vazamento de recursos
05
Validar antes de enviar — verificar os dados na aplicação antes de executar — nunca confiar apenas nas constraints do banco
