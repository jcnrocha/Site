Três Técnicas Avançadas da Programação Estruturada

🧩
Modularização
"Dividir o programa em métodos pequenos, cada um com uma única responsabilidade — o código fica legível, testável e reutilizável"
  • Princípio da Responsabilidade Única: uma função = uma tarefa
  • Máximo de 20–30 linhas por método como referência prática
  • Nome do método deve descrever o que ele faz: calcularMedia()
  • Evitar código duplicado: extrair para um método compartilhado
  • Facilita o debug: o bug fica isolado em um módulo específico
📨
Passagem de Parâmetros
"Em Java, primitivos são passados por valor (cópia); objetos e arrays são passados por referência (mesmo endereço) — saber isso evita bugs silenciosos"
  • Por valor: int, double, boolean, char — cópia passada, original intocado
  • Por referência: arrays e objetos — mesma área de memória, original pode mudar
  • Exemplo valor: dobrar(int x) → x dentro do método é cópia
  • Exemplo referência: zerar(int[] v) → altera o vetor original
  • Regra prática: quer proteger o original? Faça uma cópia antes de passar
🌀
Recursividade
"Uma função recursiva chama a si mesma com um problema menor — obrigatório ter um caso base (condição de parada) ou a pilha transborda"
  • Caso base: condição que para a recursão (ex: if(n==0) return 1;)
  • Caso recursivo: a função chama a si com n menor
  • Fatorial: fat(n) = n * fat(n-1) · caso base: fat(0) = 1
  • Pilha de chamadas: cada chamada empilha um frame na memória
  • Sem caso base → StackOverflowError

5 Recursos do Desenvolvedor Estruturado

🧩
Responsabilidade Única
Cada método faz UMA coisa — nome descreve a tarefa, fácil de testar e reutilizar isoladamente
📋
Passagem por Valor
Primitivos (int, double…) são copiados — alterar o parâmetro dentro do método não afeta o original
🔗
Passagem por Referência
Arrays e objetos compartilham o mesmo endereço de memória — alterações dentro do método afetam o original
🌀
Recursividade
Função que chama a si mesma · Sempre precisa de caso base · Útil para problemas que se dividem em subproblemas iguais
🔍
Depurador (Debug)
Breakpoint + Step Over + Step Into + Watch · Inspeciona variáveis em tempo real sem precisar de System.out.println

Código Modularizado × Código Espaguete

🧩
Código Modularizado
  • Cada método faz uma coisa: calcularMedia(), validarIdade(), imprimirRelatorio()
  • Fácil de testar: cada parte pode ser verificada isoladamente
  • Fácil de reutilizar: calcularMedia() serve em qualquer contexto
  • Fácil de depurar: o bug fica localizado em um módulo específico
  • Fácil de ler: o main descreve o fluxo em linguagem natural
  • Manutenção pontual: mudar a regra de média exige editar apenas um método
Objetivo: main() com 10–15 linhas descrevendo o fluxo; a lógica fica nos métodos
🍝
Código Espaguete
  • Tudo no main(): 150 linhas sem separação de responsabilidades
  • Código duplicado: mesma lógica copiada em 3 lugares diferentes
  • Impossível testar partes isoladas — tudo depende de tudo
  • Bug difícil de encontrar: é preciso ler o programa inteiro
  • Difícil de ler: nomes como x, aux, blocos sem nome
  • Manutenção perigosa: mudar uma linha pode quebrar outra em lugar inesperado
Sinal de alerta: método com mais de 50 linhas quase sempre tem mais de uma responsabilidade

🧩 Modularize, Trace ou Depure — Qual Técnica Usar?
⏱️ Duração: 15 minutos
👥 Agrupamento: grupos de 4 a 5 alunos
🎯 Objetivo: integrar os três temas da aula — modularização, passagem de parâmetros e recursividade — aplicando-os a situações reais e desenvolvendo raciocínio técnico sobre qualidade de código
1
Distribuição dos casos (1 min) — Cada grupo recebe 8 fichas com situações-problema. Para cada uma: (a) identificar a técnica correta e (b) escrever a solução ou explicação em linguagem Java ou texto técnico
2
Resolução em grupo (8 min) — O grupo delibera e registra. Grupos que terminam antes recebem desafio bônus: traçar fibonacci(5) completo, mostrando cada chamada recursiva e seu retorno
3
Gabarito e debate (6 min) — Professor projeta cada caso e discute com a turma; reforça os erros mais comuns: esquecer caso base, confundir = com ==, método grande demais com múltiplas responsabilidades
Situação / ProblemaTécnicaSolução / Resposta esperada
Um main() com 120 linhas calcula média, encontra maior nota e gera relatório. Como melhorar? Modularização Extrair três métodos: calcularMedia(), encontrarMaior() e gerarRelatorio(); o main() chama os três
Método recebe int x = 10, faz x = 99 dentro do método. Qual é o valor de x no chamador após o retorno? Por Valor x continua valendo 10 no chamador — primitivos são copiados; a alteração dentro do método não afeta o original
Método recebe int[] arr, faz arr[0] = 99. O vetor original muda? Por Referência Sim — arrays são objetos e passam a referência (endereço); alterações dentro do método afetam o vetor original
Trace a chamada fatorial(3) com fat(n) = n * fat(n-1), base fat(0)=1 Recursividade fat(3)=3×fat(2) → 3×2×fat(1) → 3×2×1×fat(0) → 3×2×1×1 = 6
Função recursiva sem if de parada. O que acontece quando é chamada? Recursividade StackOverflowError — a pilha de chamadas esgota a memória pois nunca há retorno
Variável tem valor errado no meio do programa mas não sabe em qual linha o valor muda. Qual ferramenta usar sem adicionar println? Debugger Colocar breakpoint na linha suspeita e usar Watch para monitorar a variável; executar em modo debug e usar Step Over para avançar linha a linha
Quer entrar dentro de um método chamado e ver o que acontece dentro dele durante o debug Debugger Usar Step Into (F11 no VS Code) em vez de Step Over — o debugger entra no corpo do método chamado
Método processarPedido() com 60 linhas valida CPF, calcula frete e envia e-mail. Está correto? Modularização Não — viola a responsabilidade única. Separar em: validarCPF(), calcularFrete() e enviarEmail()
💡 Reforço pedagógico: modularização não é opcional — é a diferença entre código que cresce de forma sustentável e código que vira um emaranhado impossível de manter. A recursividade resolve problemas elegantemente mas exige disciplina: sem caso base, não há solução — só StackOverflow. O debugger substitui dezenas de println e revela o estado real do programa a qualquer momento da execução.
