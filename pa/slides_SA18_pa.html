O ponto de partida

⚡ O que é Otimização de Código?

"Otimização prematura é a raiz de todo mal em programação."

— Donald Knuth · Cientista da computação · A frase mais citada sobre otimização
O que é Tornar o Código Mais Eficiente
Otimização é o processo de melhorar o desempenho de um programa — fazê-lo rodar mais rápido, consumir menos memória ou realizar menos operações para obter o mesmo resultado. O resultado deve ser idêntico — só a eficiência muda.
A regra de ouro Meça Primeiro — Sempre
Antes de otimizar qualquer coisa: meça o tempo de execução. A intuição engana — o trecho que "parece" lento raramente é o verdadeiro gargalo. Sem medição, você pode gastar horas otimizando o lugar errado.

Quando otimizar? Apenas quando um gargalo foi comprovado por medição, afeta a experiência do usuário e o ganho justifica o custo de legibilidade. Código lento que funciona é melhor do que código "otimizado" que ninguém entende.

A ferramenta mais simples e mais importante

⏱️ Como Medir o Tempo de Execução

Para comparar o desempenho antes e depois de uma otimização, precisamos de números reais. O método mais simples: registrar o tempo antes do trecho, registrar depois e calcular a diferença.

☕ Java — currentTimeMillis
long inicio = System.currentTimeMillis();

// trecho de código a medir
for (int i = 0; i < 100000; i++) {
  resultado += "item" + i;
}

long fim = System.currentTimeMillis();
System.out.println("Tempo: " + (fim - inicio) + "ms");
🐍 Python — time.time()
import time

inicio = time.time()

# trecho de código a medir
for i in range(100000):
  resultado += f"item{i}"

fim = time.time()
print(f"Tempo: {fim - inicio:.4f}s")
Gargalo 1 de 4

🧵 Gargalo 1 — Concatenação de String em Laço

Em Java, String é imutável — cada += dentro de um laço cria um novo objeto String na memória e descarta o anterior. Com 100.000 iterações, isso cria 100.000 objetos temporários. O resultado: memória desperdiçada e execução lenta.

✗ Antes — lento
String s = "";
for (int i = 0; i < 100000; i++) {
  s += "item" + i; // novo objeto a cada volta
}
// ~2.000ms em máquina comum
✔ Depois — rápido
StringBuilder sb = new StringBuilder();
for (int i = 0; i < 100000; i++) {
  sb.append("item").append(i);
}
String s = sb.toString();
// ~10ms — 200x mais rápido

StringBuilder acumula caracteres em um buffer interno e gera a String final só ao final — sem criar objetos temporários. Use sempre que precisar concatenar texto dentro de um laço.

Gargalo 2 de 4

📤 Gargalo 2 — Cálculo Invariante Dentro do Laço

Um cálculo invariante é aquele que produz o mesmo resultado em todas as iterações. Colocá-lo dentro do laço faz com que seja executado milhares de vezes sem necessidade — o resultado é sempre o mesmo.

✗ Antes — cálculo repetido
for (int i = 0; i < pedidos.length; i++) {
  // recalcula a taxa 50.000 vezes
  double taxa = Math.pow(aliquota, 2) * fator;
  total += pedidos[i] * taxa;
}
✔ Depois — calcule uma vez
// calcula uma única vez, antes do laço
double taxa = Math.pow(aliquota, 2) * fator;

for (int i = 0; i < pedidos.length; i++) {
  total += pedidos[i] * taxa; // usa o valor
}

Regra simples: se o valor não muda de uma iteração para outra, calcule fora do laço. Isso vale para chamadas de método, leituras de arquivo, consultas ao banco e qualquer operação custosa.

Gargalo 3 de 4

🗄️ Gargalo 3 — SELECT Dentro de Laço

Cada ida ao banco de dados envolve: abrir conexão → enviar query pela rede → aguardar resposta → receber dados. Isso leva milissegundos. Dentro de um laço de 500 clientes, isso vira 500 idas ao banco — o maior gargalo de desempenho em sistemas reais.

ERRADO
500 consultas ao banco: for (cliente : clientes) { SELECT nome WHERE id = cliente.getId(); } — Uma consulta por cliente. Com 500 clientes: 500 idas ao banco, ~2.500ms de espera.
CERTO
Uma consulta com WHERE IN: Antes do laço, busque todos de uma vez — SELECT nome, id WHERE id IN (1, 2, 3, ... 500). Guarde em um Map. Dentro do laço: consulte o Map local. Resultado: 1 ida ao banco, ~5ms.
REGRA
Nunca faça consulta ao banco dentro de um laço. Busque tudo que precisa antes do laço, armazene em uma estrutura local (List, Map, array) e consulte essa estrutura durante a iteração.
Gargalo 4 de 4

🛑 Gargalo 4 — Continuar Percorrendo Após Encontrar

✗ Antes — percorre até o fim mesmo após encontrar
int posicao = -1;
for (int i = 0; i < alunos.length; i++) {
  if (alunos[i].getId() == idBuscado) {
    posicao = i;
    // ← deveria parar aqui, mas continua...
    // percorre os 9.999 restantes sem motivo
  }
}
✔ Depois — para imediatamente ao encontrar
int posicao = -1;
for (int i = 0; i < alunos.length; i++) {
  if (alunos[i].getId() == idBuscado) {
    posicao = i;
    break; // Encerra — missão cumprida
  }
}

Quando o item está no meio da lista: sem break, uma lista de 10.000 elementos exige percorrer todos os 10.000. Com break, para ao encontrar — em média percorre apenas metade da lista.

Quando otimizar — e quando não otimizar

⚖️ Otimização Prematura × Otimização Consciente

🏃 Otimização Prematura — o erro
✗ Otimizar antes de medir — "achar" que algo é lento
✗ Sacrificar legibilidade por microganhos irrelevantes
✗ Código "esperto" que ninguém da equipe entende
✗ Otimizar trecho que executa apenas uma vez por dia
✗ Gastar horas no lugar errado do código
🎯 Otimização Consciente — o caminho
✔ 1º: escrever código correto e legível
✔ 2º: medir — onde o tempo é realmente gasto?
✔ 3º: otimizar o gargalo comprovado
✔ 4º: medir de novo — confirmar o ganho
✔ Priorizar gargalos que afetam o usuário

O fluxo correto é sempre: Medir → Identificar gargalo → Otimizar → Medir de novo. Sem o primeiro e o último passo, não é otimização — é suposição.

Visão geral das técnicas

📋 Resumo — Antes × Depois

Código lento (antes) Gargalo Código otimizado (depois)
String s = ""; for (...) s += item; Novo objeto a cada volta StringBuilder sb; sb.append(item);
for (...) { double taxa = calc(); ... } Cálculo repetido double taxa = calc(); for (...) { ... }
SELECT executado dentro de laço de 500 registros 500 idas ao banco Uma consulta com WHERE id IN (...) antes do laço
Busca percorre vetor inteiro mesmo após encontrar Iterações desperdiçadas if (encontrado) { break; } ao achar
Otimizar sem medir ("acho que está lento") Prematura Medir → Otimizar → Medir de novo

No mercado, sistemas lentos geralmente têm apenas 2 ou 3 gargalos reais — e eles quase sempre estão em laços com operações de banco, não nos algoritmos.

O que vimos hoje

📌 Resumo da Aula

⏱️ Medir Antes de Otimizar
Use System.currentTimeMillis() em Java ou time.time() em Python. Registre o tempo antes e depois — sem medição não há otimização, há suposição.
🧵 StringBuilder
Nunca concatene String com += dentro de laço. Use StringBuilder.append() — pode ser 200x mais rápido com grandes volumes.
📤 Operações Fora do Laço
Cálculos, chamadas de método e qualquer coisa que não muda entre iterações deve ficar fora do laço — calcule uma vez, use muitas.
🗄️ Nunca SELECT no Laço
Busque tudo antes do laço com WHERE IN, guarde em Map/List local e consulte a estrutura local. 500 clientes = 1 consulta, não 500.

A lição mais importante: código legível vem primeiro. Otimize apenas o que foi medido e comprovado como gargalo — otimização prematura é armadilha.
