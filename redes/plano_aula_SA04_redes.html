Arquiteturas de Armazenamento — Local · Rede · Nuvem

💾
Armazenamento Local
"Conectado diretamente ao computador — simples, rápido e sem compartilhamento nativo"
  • DAS — Direct Attached Storage · conectado fisicamente ao computador por SATA, USB, eSATA ou SAS · sem intermediário de rede · exemplos: HD interno, SSD, HD externo USB, fita magnética
  • Acesso exclusivo: apenas o computador diretamente conectado acessa os dados
  • Protocolo: SATA, USB 3.0/3.2, Thunderbolt — protocolos de storage local, não de rede
  • Vantagem: latência muito baixa · custo simples · fácil configuração
  • Desvantagem: não compartilha nativamente · expansão física limitada · backup depende do gestor local
  • Uso típico: workstations, laptops, servidores de aplicação de alto desempenho com storage local
🗄️
Armazenamento em Rede
"Acessado pela rede local — múltiplos computadores compartilham o mesmo storage ao mesmo tempo"
  • NAS — Network Attached Storage · dispositivo dedicado ao armazenamento de arquivos na rede local · acessa via protocolo de arquivo: NFS (Linux/macOS), SMB/CIFS (Windows) · aparece como unidade de rede mapeada no computador · exemplos: Synology, QNAP, Windows Server com compartilhamento
  • Acesso em nível de arquivo — você manipula pastas e arquivos, como em qualquer diretório
  • Uso típico: compartilhamento de documentos em escritório, backup centralizado, home media server
  • SAN — Storage Area Network · rede dedicada de alta velocidade para storage de blocos · conecta servidores ao storage por Fibre Channel (FC) ou iSCSI (TCP/IP) · o servidor "enxerga" o storage como se fosse um HD local · exemplos: EMC VNX, NetApp, IBM Storage
  • Acesso em nível de bloco — o sistema operacional formata e gerencia o storage como se fosse físico
  • Uso típico: banco de dados de alta performance, virtualização (VMware, Hyper-V), aplicações críticas de negócio
☁️
Armazenamento em Nuvem
"Acessado pela internet — escalável, sem hardware próprio, pago pelo uso"
  • Nuvem pública · infraestrutura do provedor, compartilhada entre clientes · paga pelo uso (pay-as-you-go) · exemplos: Amazon S3, Google Cloud Storage, Azure Blob, Google Drive, OneDrive, Dropbox
  • Nuvem privada · infraestrutura dedicada para uma organização · maior controle e segurança · pode ser on-premises (dentro da empresa) ou hosted · exemplos: VMware Private Cloud, OpenStack, Microsoft Azure Stack
  • Nuvem híbrida · combinação de on-premises + nuvem pública · dados sensíveis ficam locais; dados menos críticos vão para a nuvem · permite burst (expansão automática quando necessário)
  • Modelo de armazenamento como serviço (STaaS): sem compra de hardware, manutenção pelo provedor, acesso de qualquer lugar com internet
  • Desvantagem: depende de conectividade · latência maior que local · custo pode crescer com o volume

NAS × SAN — A Distinção Mais Confusa do Armazenamento em Rede

🗂️
NAS
Acesso por arquivo · Ethernet comum
  • Acesso em nível de arquivo — o sistema operacional solicita um arquivo pelo nome e o NAS o entrega
  • Usa a rede Ethernet comum (LAN) com TCP/IP
  • Protocolos: SMB/CIFS (Windows) · NFS (Linux/macOS)
  • Aparece como unidade de rede mapeada no explorador de arquivos
  • Custo acessível — hardware simples, sem rede dedicada
  • Ideal para: compartilhamento de arquivos em escritório, backup centralizado, streaming de mídia em rede local
NAS: o servidor pede um arquivo → o NAS pesquisa e entrega o arquivo completo
⚡
SAN
Acesso por bloco · rede dedicada
  • Acesso em nível de bloco — o servidor gerencia o filesystem diretamente, como se fosse um HD local
  • Usa rede dedicada: Fibre Channel (velocidades de 8/16/32 Gbps) ou iSCSI (SCSI encapsulado em TCP/IP)
  • O servidor formata e monta o storage como se fosse um volume local (ex.: /dev/sdb no Linux)
  • Latência muito baixa — próximo ao desempenho de storage local
  • Custo elevado — hardware especializado, switches FC, cabos ópticos
  • Ideal para: banco de dados Oracle/SQL, virtualização VMware, aplicações que exigem IOPS altos
SAN: o servidor controla o filesystem e pede blocos brutos → a SAN entrega blocos de dados

Modelos de Nuvem — Pública, Privada e Híbrida

☁️ Nuvem Pública
  • Infraestrutura do provedor — AWS, Google Cloud, Azure
  • Pago conforme o uso (pay-as-you-go) — sem CAPEX
  • Escalabilidade instantânea — adiciona TB em segundos
  • Responsabilidade compartilhada: provedor cuida da infraestrutura; cliente cuida dos dados
  • Exemplos de uso: backup offsite, sites e apps web, armazenamento de dados frios (S3 Glacier)
🔒 Nuvem Privada
  • Infraestrutura dedicada — dentro da empresa (on-premises) ou em datacenter exclusivo
  • Total controle sobre dados e segurança — exigência de compliance (LGPD, HIPAA, PCI-DSS)
  • Custo fixo alto — CAPEX + equipe de TI
  • Exemplos: OpenStack, VMware vSphere privado, Azure Stack Hub
  • Uso típico: dados médicos, financeiros, governamentais ou estratégicos
🔀 Nuvem Híbrida
  • Combina on-premises + nuvem pública de forma integrada
  • Dados sensíveis ficam locais; dados de acesso público ou frios vão para a nuvem
  • Cloud burst: durante picos de demanda, cargas transbordam para a nuvem automaticamente
  • Exemplos: Azure Arc, AWS Outposts, Google Anthos
  • Uso típico: empresas com dados regulados que também querem escalabilidade da nuvem pública
💡 Conexão com H1 (SA 03): as unidades de medida aprendidas na SA 03 são fundamentais aqui. Um NAS doméstico de 4 TB a 1 Gbps de Ethernet tem throughput máximo de 125 MB/s — suficiente para streaming de vídeo 4K (≈ 25 Mbps). Uma SAN Fibre Channel de 16 Gbps suporta 2 GB/s — necessário para bank de dados que exigem centenas de milhares de IOPS. A escolha da arquitetura de armazenamento é sempre uma equação de capacidade, velocidade e custo.

🎮 Dinâmica Qual Armazenamento Usar?
🏢 Qual Armazenamento Usar? — Escolha a Arquitetura para Cada Cenário
⏱️ Duração: 16 minutos
👥 Agrupamento: grupos de 4 a 5 alunos
🎯 Objetivo: aplicar os conceitos de DAS, NAS, SAN e Cloud a situações reais, desenvolvendo o raciocínio de escolha de arquitetura. O aluno precisa justificar considerando custo, desempenho, compartilhamento e criticidade dos dados
1
Distribuição dos cartões de cenário (1 min) — Cada grupo recebe fichas com 6 cenários de empresas ou situações reais. A tarefa: indicar a arquitetura mais adequada (DAS / NAS / SAN / Cloud pública / Cloud híbrida) e escrever a justificativa em 1 ou 2 frases. Respostas sem justificativa não pontuam
2
Análise e discussão em grupo (10 min) — O grupo debate cada cenário. Atenção: alguns cenários admitem mais de uma resposta válida — o grupo precisa argumentar qual é a melhor considerando os critérios fornecidos. O professor circula entre os grupos para ouvir os argumentos
3
Comparação e gabarito comentado (5 min) — Professor revela a resposta de cada cenário e discute as justificativas. Destaque especial para grupos que argumentaram bem em cenários com mais de uma resposta aceitável — o processo de raciocínio vale tanto quanto a resposta correta
CenárioResposta idealNível
Um estudante de programação quer guardar seus projetos no computador pessoal e acessá-los apenas do próprio notebook. Não precisa compartilhar com ninguém DAS — uso exclusivo, sem necessidade de rede Básico
Um escritório de contabilidade com 8 funcionários precisa que todos acessem e editem os mesmos arquivos de clientes. Orçamento limitado, sem equipe de TI especializada NAS — compartilhamento simples, custo baixo, Ethernet existente Básico
Uma startup com equipe remota em 3 países precisa que todos acessem documentos de qualquer lugar, sem investimento em hardware próprio. Dados não são sensíveis Cloud Pública — acesso global, sem hardware, escalável e pago por uso Médio
Um hospital precisa armazenar prontuários eletrônicos de pacientes. Exige altíssima disponibilidade, conformidade com regulação de dados médicos (LGPD) e acesso de múltiplos sistemas internos simultaneamente NAS ou Cloud Privada — controle total, conformidade regulatória, sem dados sensíveis em nuvem pública Médio
Um banco precisa de storage para seu banco de dados de transações. Exige latência abaixo de 1 ms, 100.000 IOPS e alta disponibilidade com failover automático SAN — acesso por bloco, Fibre Channel, IOPS e latência impossíveis de atingir com NAS ou cloud Avançado
Uma rede de varejo com 200 lojas precisa manter dados de vendas localmente em cada loja por conformidade fiscal, mas quer consolidar relatórios gerenciais em uma plataforma central acessível pelos gestores Cloud Híbrida — dados fiscais locais (on-premises), relatórios na nuvem pública; burst para picos de acesso Avançado
💡 Critério de ouro para a decisão: (1) precisa compartilhar? Se não → DAS; se sim → próxima pergunta; (2) precisa de performance de banco de dados ou VM? → SAN; se não → próxima pergunta; (3) os dados são sensíveis/regulados? Se sim → NAS ou Cloud Privada; se não → Cloud Pública pode ser mais eficiente. A Cloud Híbrida é a resposta para quem tem os dois tipos de dado na mesma empresa.
